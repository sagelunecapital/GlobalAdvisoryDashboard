### DEC-2026-04-18-02 — Divergence anchor = prior period's swing high/low

**Status: COMPLIANT**

The anchor algorithm explicitly prevents yesterday's same-run high from becoming the anchor:

- **Bearish anchor:** Global trough in the 90-day prior window is identified first. The anchor is the pre-trough slice maximum — i.e., the peak before the correction, not any post-trough date. On Day 2 of a run, the Day 1 row joins the prior window, but the trough remains the global minimum (6,500 in T9), so the pre-trough peak is still the same Jan 5 / 7,002 anchor.

- **Bullish anchor:** Global peak in the window is identified first. The anchor is the post-peak slice minimum.

T9 explicitly validates this: Day 2 evaluation still returns `"7002"` in the message despite Day 1's 7,026 being in the prior window.

## Minor Observations (non-blocking)

1. **No NaN-anchor test:** The DATA_GAP guards for MMTH NaN at anchor date (lines 172–177, 196–201) have no dedicated test. The code is correct and the guard is unreachable without a null MMTH in the DB, but a future test would increase coverage and protect against refactor regression.

2. **`post_peak_df` empty check dead code:** At line 186–187, the `if post_peak_df.empty` fallback uses `window_df["spx_daily_high"].idxmin()`. However, since `post_peak_df = window_df[window_df["date"] >= peak_date]` and `peak_date` is the date of `idxmax()`, `post_peak_df` always contains at least the peak row itself. The branch is unreachable in practice. No functional issue — the fallback would produce correct behavior if triggered.

3. **`prior_low_date` unused in message:** The bullish divergence message references `prior_low_date` (line 222) correctly — this is fine, just noting it mirrors bearish message symmetry.

None of these observations affect the PASS verdict.
