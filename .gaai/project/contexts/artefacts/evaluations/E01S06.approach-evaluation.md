# Approach Evaluation — E01S06: Initialize and Maintain Historical Indicator Database

## Problem Statement

The project requires a persistent, append-only database of four daily indicator values (SPX daily high, SPX 12d EMA, SPX 25d EMA, MMTH) seeded with at least one year of historical data. This is the foundation story — no language, framework, or database has been chosen yet. Three independent technology decisions must be evaluated before implementation begins: (1) the programming language/runtime, (2) the database technology, and (3) the strategy for fetching historical SPX and MMTH data. The project is an internal tool for 1-2 fund managers; simplicity and maintainability take priority over scalability.

## Decision Point 1: Programming Language / Runtime

### Evaluation Criteria

| C4 | Community maturity for data/finance workloads | important | Industry standard signal |
| C5 | Cold-start / operational overhead — no runtime install complexity | nice-to-have | Project context: internal tool, not a hosted service |

### Approaches Identified

#### Approach A — Python 3.x

**Mechanism:** Interpreted, dynamically typed language. Wide adoption in data science and finance. The standard data stack (pandas, numpy, yfinance, ta-lib) is mature, well-documented, and directly addresses all requirements. SQLite is supported via the built-in `sqlite3` module (no third-party dependency).

**Evidence:** Python is the de-facto standard for financial data processing scripts, quant research pipelines, and internal analytics tools. `yfinance` (Yahoo Finance wrapper) is the most widely used open-source library for fetching OHLC historical data for equities indices including `^GSPC`. `pandas` provides EMA computation via `DataFrame.ewm()`. Projects such as Backtrader, Zipline, and QuantConnect use Python throughout. PyPI has >50 actively maintained financial data libraries as of 2025.

**Limitations:** Slower than compiled languages for CPU-bound workloads, but this is irrelevant for a daily-batch fetch of ~260 rows/year. Requires Python 3.x to be installed on the target machine (standard on macOS/Linux; available on Windows).

#### Approach B — JavaScript / Node.js

**Mechanism:** Server-side JavaScript runtime. Suitable for I/O-bound tasks. Has `better-sqlite3` for synchronous SQLite access and some financial data libraries (`yahoo-finance2`).

**Evidence:** `yahoo-finance2` is a maintained Node.js wrapper for Yahoo Finance with OHLC support. `better-sqlite3` is the recommended synchronous SQLite driver for Node. Used in some fintech web applications and dashboards (e.g., portfolio trackers built on Next.js).

**Limitations:** The financial data library ecosystem is significantly thinner than Python's. EMA computation requires either a third-party library (e.g., `technicalindicators`) or manual implementation — neither is as battle-tested as `pandas.ewm()`. MMTH data availability via Node libraries is unconfirmed. The async programming model (Promises, async/await) adds ceremony for a sequential batch-load script.

#### Approach C — Go

**Mechanism:** Compiled, statically typed language. Fast, low memory footprint. SQLite accessible via `mattn/go-sqlite3` (CGO) or `modernc.org/sqlite` (pure Go).

**Evidence:** Go is used in production data pipelines at companies like Cloudflare and Uber. Financial data libraries exist but are sparse; no widely adopted Go equivalent of `yfinance` or `pandas` exists as of 2025.

**Limitations:** No mature Go library for fetching Yahoo Finance OHLC data — custom HTTP parsing would be required. EMA computation would need manual implementation. Significantly higher implementation effort for no operational benefit given the internal-tool context. CGO dependency for SQLite adds build complexity.

| C2: SQLite integration | PASS — built-in `sqlite3` module, zero extra deps | PASS — `better-sqlite3` is mature and widely used | PARTIAL — `modernc.org/sqlite` is pure Go but less battle-tested than Python stdlib |

| C5: Operational overhead | PASS — pre-installed on macOS/Linux; one-line install on Windows | PASS — Node.js widely available | PARTIAL — requires Go toolchain; CGO adds build step |

### Trade-offs — DP1

#### Python 3.x
- **Gains:** Zero-friction access to the exact libraries needed (yfinance, pandas, sqlite3). EMA computation is one line. Community support is dense.
- **Costs:** Slightly slower than compiled alternatives (irrelevant at this scale). Requires Python installed.
- **Lock-in:** Low — Python scripts are easy to rewrite or extend; no framework lock-in at this stage.

#### JavaScript / Node.js
- **Gains:** Strong web ecosystem for when a UI layer is added (later stories). Unified language if the dashboard UI is also JS-based.
- **Costs:** Thinner financial library ecosystem now. EMA needs a third-party or manual solution. MMTH fetch path is unproven in Node.
- **Lock-in:** Medium — if UI stories choose a different frontend framework, the Node backend advantage dissolves.

#### Go
- **Gains:** Fast, self-contained binary; no runtime dependency.
- **Costs:** Highest implementation effort; no library for the core data fetch requirement.
- **Lock-in:** Medium — Go's type system makes refactoring safe but initial cost is high for this problem.

## Decision Point 2: Database Technology

### Evaluation Criteria

| # | Criterion | Weight | Source |
|---|-----------|--------|--------|
| C1 | Zero-ops overhead — no server process, no DBA, no provisioning | must-have | Project context: small internal tool, 1-2 users |
| C2 | ACID guarantees — especially for append-only writes (AC3, AC4) | must-have | Story AC3, AC4: append-only, no corruption under failure |
| C3 | Persistence across restarts (AC5) | must-have | Story AC5 |
| C4 | Python integration — stable, well-maintained driver | must-have | Dependent on DP1 decision |
| C5 | Schema flexibility — ability to add columns later if new indicators are added | nice-to-have | Future extensibility |
| C6 | Queryable — allows ad-hoc date-range queries for downstream divergence detection (E01S02) | important | Downstream story dependency |

### Approaches Identified

#### Approach A — SQLite

**Mechanism:** Embedded relational database stored in a single `.db` file. ACID-compliant via WAL mode. Requires no server. Accessed via Python's built-in `sqlite3` module. Supports full SQL including date-range queries.

**Evidence:** SQLite is the world's most deployed database engine (used in iOS, Android, Firefox, Python stdlib). The Python `sqlite3` module is part of the standard library since Python 2.5. Used in production by small-to-medium internal tools, analytics dashboards, and data pipelines where a full server is unjustified. SQLite's WAL mode enables concurrent reads with single-writer guarantees. As of SQLite 3.40+ (2022), WAL2 mode further reduces checkpoint contention.

**Limitations:** Single-writer at a time (irrelevant for a daily-batch tool with 1-2 users). File must be accessible from the machine running the dashboard — not suitable for distributed multi-node setups (not a constraint here). Maximum practical size ~1TB; daily indicator data for 1 year = ~260 rows, negligible.

#### Approach B — PostgreSQL

**Mechanism:** Full client-server RDBMS. Requires a running PostgreSQL server process. Python access via `psycopg2` or `asyncpg`. Full SQL, ACID, high concurrency.

**Evidence:** PostgreSQL is the most feature-complete open-source RDBMS. Used for large-scale applications. Can be run locally via Docker or a managed service (Supabase, Neon, AWS RDS).

**Limitations:** Requires a running server process — adds ops overhead (install, configure, maintain, backup). For 260 rows/year of indicator data, this is significant over-engineering. Cold-start on a new machine requires server setup. Contradicts the "simplicity and maintainability" constraint from project context.

#### Approach C — CSV / Flat Files

**Mechanism:** One CSV file per indicator or one combined CSV with date + indicator columns. Append is a file write. Python access via `csv` module or `pandas.read_csv()`.

**Evidence:** Used widely in data science notebooks and research scripts. Simple to inspect and debug. Human-readable.

**Limitations:** No ACID guarantees — a crash mid-write leaves a corrupt or partial file (directly violates AC6: AC6 requires a clearly retryable state with no silent partial corruption). No query language — downstream divergence detection (E01S02) would need to load the entire file into memory and filter in Python. No schema enforcement — typos or format changes silently corrupt the dataset. Appending to CSV in a multi-line transaction is not atomic.

### Comparison Matrix — DP2

| Criterion | SQLite | PostgreSQL | CSV / Flat Files |
|-----------|--------|-----------|-----------------|
| C1: Zero-ops overhead | PASS — single file, no server, no installation | FAIL — requires running server process, installation, config | PASS — plain files, no setup |
| C2: ACID guarantees | PASS — WAL mode provides full ACID; transactions wrap appends | PASS — full ACID, mature transaction support | FAIL — no atomic writes; crash mid-append corrupts file |
| C3: Persistence across restarts | PASS — durable file | PASS — durable if server running | PASS — durable file |
| C4: Python integration | PASS — built-in `sqlite3`, zero deps | PARTIAL — requires `psycopg2` install + server running | PASS — built-in `csv` module or pandas |
| C5: Schema flexibility | PASS — ALTER TABLE for column additions | PASS — full DDL support | PARTIAL — manual format migration required |
| C6: Queryable | PASS — full SQL, indexed date queries | PASS — full SQL | FAIL — full-file load required for every query |

### Trade-offs — DP2

#### SQLite
- **Gains:** ACID with zero ops overhead. Full SQL for downstream queries. Single-file portability. Built-in Python support.
- **Costs:** Single-writer constraint (irrelevant here). File path must be managed.
- **Lock-in:** Low — SQLite files are portable; migrating to PostgreSQL later requires a one-time export/import.

#### PostgreSQL
- **Gains:** Highest concurrency and feature set. Cloud-native deployment options.
- **Costs:** Server process required. Contradicts simplicity constraint. Over-engineered for 260 rows/year.
- **Lock-in:** Medium — schema and SQL compatible with SQLite migration, but ops overhead is permanent.

#### CSV / Flat Files
- **Gains:** Zero dependencies. Human-readable for manual inspection.
- **Costs:** Fails ACID requirement (AC6). No query capability. No schema enforcement.
- **Lock-in:** Low technically, but migration to a real DB requires a one-time data transformation.

## Decision Point 3: Data Fetching Strategy for SPX and MMTH

### Evaluation Criteria

| # | Criterion | Weight | Source |
|---|-----------|--------|--------|
| C1 | Historical data depth — 1+ year of daily OHLC for SPX and MMTH | must-have | Story AC1 |
| C2 | SPX daily high availability — must retrieve daily high specifically, not just close | must-have | DEC-2026-04-18-01 |
| C3 | No login / account required for historical data | must-have | Confirmed data sources note |
| C4 | Library reliability — not a fragile scraper dependent on HTML structure | important | Operational risk |
| C5 | MMTH availability — % stocks above 200d MA, confirmed accurate | must-have | Confirmed data sources note |

### Approaches Identified

#### Approach A — `yfinance` Python library (Yahoo Finance wrapper)

**Mechanism:** `yfinance` wraps Yahoo Finance's undocumented API to fetch OHLC data for equities and indices. For SPX, the ticker is `^GSPC`. For MMTH, the ticker is `^MMTH` (if available — see Open Questions). Supports `download(period='2y', interval='1d')` which returns a DataFrame with Open, High, Low, Close, Volume. The `High` column provides SPX daily high directly.

**Evidence:** `yfinance` is the most widely used Python library for fetching historical financial data as of 2025, with >15,000 GitHub stars and active maintenance. Yahoo Finance provides multiple years of daily OHLC for major indices including `^GSPC`. `^GSPC` daily high data confirmed accurate in project context (Apr 15 high = 7,026.24). SPX EMA computation uses the `High` column directly. `yfinance` is used in hundreds of open-source financial analysis projects (QuantLib integrations, backtesting frameworks, research notebooks).

**Limitations:** Yahoo Finance's API is undocumented and has historically changed without notice, breaking `yfinance` until a fix is released. MMTH (`^MMTH`) availability on Yahoo Finance is not confirmed in the context bundle — this is an open question. Rate limiting applies for large bulk downloads; a one-year initial load (~260 rows) is well within practical limits. Not suitable for real-time data (15-minute delay), which is acceptable for a daily dashboard.

#### Approach B — EODData direct HTTP fetch for MMTH + yfinance for SPX

**Mechanism:** Use EODData's web endpoint for MMTH historical data (confirmed source in context bundle: `INDEX/MMTH`). EODData provides downloadable CSV exports for index data. Combine with `yfinance` for SPX. Two fetch paths — one per data source.

**Evidence:** EODData confirmed accurate for MMTH in project context: Apr 15 = 54.99, Apr 16 = 55.45, Apr 17 = 58.54. EODData provides historical OHLC data for market breadth indicators including MMTH. Direct CSV download requires parsing HTML or using a registered download URL. Some EODData data requires a (free or paid) account.

**Limitations:** Requires two separate fetch implementations. EODData may require account registration for full historical CSV download — contradicts C3 (no login required). HTML scraping is fragile. Account requirement makes this harder to set up on a new machine.

#### Approach C — Direct HTTP scraping of Barchart / Investing.com

**Mechanism:** Use `requests` + `BeautifulSoup` to scrape historical data tables from Barchart (`$MMTH`) or Investing.com (SPX). Parse HTML tables to extract OHLC rows.

**Evidence:** Barchart provides `$MMTH` current and recent data. Investing.com provides SPX historical OHLC tables going back years. Both are cited as sources in the context bundle.

**Limitations:** Both sites employ anti-scraping measures (Cloudflare, JS rendering). Barchart requires login for full MMTH history (noted in context bundle: "interactive chart requires login for full history"). HTML structure changes break scrapers silently. Fragile, high maintenance. This approach is explicitly the least reliable among the three.

### Comparison Matrix — DP3

| Criterion | yfinance (SPX + attempt MMTH) | EODData HTTP + yfinance | Direct Scraping |
|-----------|------------------------------|------------------------|----------------|
| C1: Historical depth 1+ year | PASS for SPX; UNCERTAIN for MMTH `^MMTH` | PASS for SPX (yfinance); PARTIAL for MMTH (account may be needed) | PARTIAL — Investing.com yes for SPX; Barchart requires login for MMTH history |
| C2: SPX daily high | PASS — `High` column available directly | PASS — yfinance High column | PARTIAL — table scraping may not isolate High cleanly |
| C3: No login required | PASS for SPX; UNCERTAIN for MMTH | PARTIAL — EODData may require registration | FAIL — Barchart requires login for full MMTH history |
| C4: Library reliability | PASS — maintained library, not HTML-dependent | PARTIAL — yfinance yes; EODData scrape fragile | FAIL — fragile HTML scraping, JS rendering issues |
| C5: MMTH availability | UNCERTAIN — `^MMTH` ticker needs verification at runtime | PASS (confirmed source) but may need account | FAIL — full history requires login |

### Trade-offs — DP3

#### yfinance (SPX + MMTH attempt)
- **Gains:** Single library, single fetch path. Well-maintained. SPX confirmed. If `^MMTH` works on Yahoo Finance, this is the simplest possible approach.
- **Costs:** `^MMTH` availability is an open question. If it fails, a fallback fetch path must be implemented.
- **Lock-in:** Low — `yfinance` is easily swapped for another library; the DB schema is independent.

#### EODData HTTP + yfinance
- **Gains:** MMTH source confirmed accurate. SPX via yfinance confirmed.
- **Costs:** Two fetch paths. EODData may require account registration, complicating initial setup.
- **Lock-in:** Medium — EODData fetch code is custom; if EODData changes its URL structure, the scraper breaks.

#### Direct Scraping
- **Gains:** None that outweigh the costs.
- **Costs:** Fragile. Login required for full MMTH history. High maintenance risk.
- **Lock-in:** High (fragile dependency on site structure).

## Open Questions

1. **MMTH via yfinance:** Is `^MMTH` available as a downloadable ticker on Yahoo Finance with 1+ year of daily history? This must be verified at the start of the Delivery Agent's execution — a one-line `yfinance.download('^MMTH', period='2y', interval='1d')` check. If it returns data, Approach A (single library) is confirmed. If not, the delivery agent must implement the EODData fetch path as fallback.

2. **EODData account requirement:** Does EODData's MMTH CSV export require a registered account for 1+ year of history? If so, Approach B requires either (a) the operator providing credentials, or (b) a different confirmed no-login source for MMTH history.

3. **EMA warm-up data:** Both Approaches A and B must fetch more than 1 year of SPX data — specifically, the plan calls for a 25-day warm-up buffer (fetch 1 year + 25 extra trading days) so that EMA values at the 1-year boundary are accurately computed from sufficient prior data. `yfinance` supports `period='2y'` to ensure this.
