# Approach Evaluation — E03S01: Fetch and Persist 32 Macro Indicators

## Problem Statement

E03S01 introduces a new `macro_indicators` table and a new set of fetch modules covering 32
indicators across five categories. Four design decisions must be resolved before implementation
begins:

1. How to access the FRED API (new dependency for this project)
2. Exact source assignments for all 32 indicators (yfinance tickers, FRED series IDs, GDPNow URL)
3. GDPNow fetch method from the Atlanta Fed
4. Schema for the new `macro_indicators` table

The existing `indicators` table (E01) must not be modified. This evaluation covers only the new
`macro_indicators` table and the fetch infrastructure for E03.

---

## Decision Point 1: FRED API Access Method

### Context

Two indicators require FRED as the authorized secondary source:
- 5-Year Breakeven Inflation Rate (FRED series: T5YIE)
- 2-Year Breakeven Inflation Rate (FRED series: T2YIE)

A `FRED_API_KEY` environment variable is already confirmed as required. The question is whether
to access FRED via direct `requests` HTTP calls or via a dedicated Python library.

### Evaluation Criteria

| # | Criterion | Weight |
|---|-----------|--------|
| C1 | Zero additional dependencies — prefer what is already in requirements.txt | important |
| C2 | Sufficient for fetching simple time series by series ID | must-have |
| C3 | Reliability — stable API surface not subject to breaking changes | must-have |
| C4 | Ease of error handling — clear HTTP status codes and error bodies | important |
| C5 | Maintenance burden — ongoing compatibility risk | important |

### Approaches Identified

#### Approach A — `requests` library (direct FRED REST API)

**Mechanism:** Call the FRED REST API directly using the `requests` library (already in
`requirements.txt`). The FRED API endpoint for observation data is:

```
GET https://api.stlouisfed.org/fred/series/observations
  ?series_id=T5YIE
  &api_key=<FRED_API_KEY>
  &file_type=json
  &observation_start=<date>
  &observation_end=<date>
```

Response is JSON with an `observations` array of `{date, value}` objects. Straightforward to
parse with standard Python.

**Evidence:** The FRED REST API is a stable, versioned, publicly documented API
(https://fred.stlouisfed.org/docs/api/fred/). FRED has maintained backward compatibility on the
v1 API since 2012. The `requests` library is already a project dependency. Fetching observations
requires exactly one endpoint. `requests.get()` with query parameters and `response.raise_for_status()`
provides clean HTTP error handling. Used widely in data pipeline projects that consume FRED data
without a dedicated client library.

**Limitations:** Requires manually constructing the query parameters. Date parsing of the response
is manual (though trivial — FRED returns ISO-8601 dates). No automatic retry or pagination
handling built in (FRED paginates at 100,000 observations, which is irrelevant for 52-week or
12-month series).

#### Approach B — `fredapi` Python library

**Mechanism:** Third-party Python library wrapping the FRED REST API. Usage:
```python
from fredapi import Fred
fred = Fred(api_key=os.environ["FRED_API_KEY"])
series = fred.get_series("T5YIE", observation_start="2024-01-01")
```

Returns a `pandas.Series` directly.

**Evidence:** `fredapi` is a popular FRED client (GitHub: mortada/fredapi, ~1k stars). Returns
data as pandas Series, which aligns with the existing project pattern of returning Series/DataFrames
from fetch modules.

**Limitations:** Adds a new dependency not currently in `requirements.txt`. The project fetches
exactly 2 FRED series — the overhead of adding and maintaining a library for 2 API calls is
disproportionate. `fredapi` is a thin wrapper around the same FRED REST API that `requests` can
call directly. Dependency adds version-pinning and compatibility risk over the project lifetime.
The `fredapi` library has had periods of maintenance inactivity.

#### Approach C — `pandas_datareader`

**Mechanism:** `pandas_datareader` provides a unified interface for many data sources including
FRED:
```python
import pandas_datareader as pdr
series = pdr.get_data_fred("T5YIE", start=start_date)
```

**Evidence:** `pandas_datareader` is a mature library with FRED support.

**Limitations:** Heavy dependency — `pandas_datareader` covers many sources (OECD, World Bank,
EDGAR, Quandl, etc.) that are irrelevant to this project. Installing it for 2 FRED series adds
~800KB of unnecessary transitive dependencies. The FRED reader in `pandas_datareader` internally
calls the same REST API. Maintenance complexity vs. benefit is unfavorable for this narrow use case.

### Comparison Matrix — DP1

| Criterion | requests (direct) | fredapi | pandas_datareader |
|-----------|------------------|---------|-------------------|
| C1: Zero new dependencies | PASS — already in requirements.txt | FAIL — new dependency | FAIL — new heavyweight dependency |
| C2: Sufficient for series fetch | PASS — one endpoint covers all needs | PASS — convenient Series output | PASS — convenient but over-featured |
| C3: API stability | PASS — FRED REST API v1 stable since 2012 | PASS — stable (thin wrapper) | PARTIAL — reader occasionally lags FRED API changes |
| C4: Error handling | PASS — HTTP status + `raise_for_status()` | PARTIAL — raises generic exceptions | PARTIAL — error messages less clear |
| C5: Maintenance burden | PASS — no additional library to maintain | PARTIAL — infrequent maintenance periods | PARTIAL — broad scope increases drift risk |

### Decision: Approach A — `requests` (direct FRED REST API)

**Rationale:** `requests` is already a project dependency. The FRED REST API is stable,
well-documented, and requires exactly one endpoint for all observation fetches. Adding `fredapi`
or `pandas_datareader` for 2 series calls introduces new dependencies disproportionate to the
benefit. Direct HTTP access provides the same data with full control over error handling,
consistent with the project's existing pattern in `mmth.py` (which uses `requests` for EODData).

## Decision Point 2: Ticker and Series ID Assignments for All 32 Indicators

### Evaluation Criteria

| # | Criterion | Weight |
|---|-----------|--------|
| C1 | Data availability — source confirmed to provide the series | must-have |
| C2 | Historical depth — 52 weekly or 12 monthly observations minimum | must-have |
| C3 | Alignment with authorized sources (yfinance primary, FRED secondary, Atlanta Fed for GDPNow) | must-have |
| C4 | Spread computation (2s10s, 10Y-3M) derivable from fetched yields if no direct feed | must-have |
| C5 | Continuous front-month for futures | must-have |

### Source Assignments

#### Category 1: Inflation (7 indicators) — monthly-released

Monthly-released indicators use the most recently released value as the current weekly
observation (AC2). FRED is the authoritative source for CPI, PCE, and PPI releases. FRED series
IDs are stable and the series have decades of history.

| Indicator | Indicator ID | Source | Series ID / Ticker | Notes |
|-----------|-------------|--------|-------------------|-------|
| CPI YoY | CPI_YOY | FRED | CPIAUCSL | Monthly, not seasonally adj. available; use CPIAUCSL (SA) and compute YoY: ((current - 12mo ago) / 12mo ago) * 100. Alternatively, FRED series CPIAUCSL_PC1 provides YoY % directly but may lag. Preferred: CPIAUCSL and compute YoY in fetch layer. |
| Core CPI YoY | CORE_CPI_YOY | FRED | CPILFESL | CPI less food and energy; compute YoY from monthly levels |
| PCE YoY | PCE_YOY | FRED | PCEPI | Personal Consumption Expenditures Price Index (monthly); compute YoY from levels |
| Core PCE YoY | CORE_PCE_YOY | FRED | PCEPILFE | Core PCE (ex food and energy); compute YoY from levels |
| PPI YoY | PPI_YOY | FRED | PPIACO | PPI All Commodities; compute YoY from levels. Note: PPIACO is a composite; FRED also has PPIFIS (finished goods). PPIACO is the broadest and most cited headline PPI. |
| 5Y Breakeven Inflation | BREAKEVEN_5Y | FRED | T5YIE | 5-Year Breakeven Inflation Rate; daily frequency; available directly as %, no YoY computation needed |
| 2Y Breakeven Inflation | BREAKEVEN_2Y | FRED | T2YIE | 2-Year Breakeven Inflation Rate; daily frequency; available directly as % |

**Note on YoY computation:** For CPI, Core CPI, PCE, Core PCE, and PPI — FRED provides the
price index level, not the YoY % directly (CPIAUCSL_PC1 provides YoY but is released with
additional lag). The fetch module will compute YoY as: `(current_month / same_month_12_periods_ago - 1) * 100`.
This requires fetching at least 24 months of history to compute 12 valid YoY observations.
Breakeven rates are already in % terms and do not require YoY computation.

#### Category 2: Yields (7 indicators) — daily, yfinance

Treasury yields and spreads are available via yfinance using Yahoo Finance index tickers. The
Effective Federal Funds Rate is a FRED series updated daily.

| Indicator | Indicator ID | Source | Ticker / Series ID | Notes |
|-----------|-------------|--------|-------------------|-------|
| US 2Y Treasury Yield | US_2Y_YIELD | yfinance | ^IRX (13-week) — see note | yfinance provides ^IRX (3-month T-bill) and ^FVX (5Y). For 2Y: use FRED DGS2 as primary since yfinance ^IRX = 3-month, not 2-year. See below. |
| US 5Y Treasury Yield | US_5Y_YIELD | yfinance | ^FVX | Yahoo Finance 5-Year Treasury Note Yield; confirmed available |
| US 10Y Treasury Yield | US_10Y_YIELD | yfinance | ^TNX | Yahoo Finance 10-Year Treasury Note Yield; confirmed available |
| US 30Y Treasury Yield | US_30Y_YIELD | yfinance | ^TYX | Yahoo Finance 30-Year Treasury Bond Yield; confirmed available |
| 2s10s Spread | SPREAD_2S10S | computed | US_10Y_YIELD - US_2Y_YIELD | Computed from fetched yield series; stored as a derived series |
| 10Y-3M Spread | SPREAD_10Y3M | computed | US_10Y_YIELD - US_3M_YIELD | Requires fetching 3M yield; use ^IRX (yfinance) or FRED DGS3MO for 3M |
| Effective Fed Funds Rate | EFFR | FRED | EFFR | Effective Federal Funds Rate; daily frequency from FRED; stable and authoritative |

**Yield source clarification — 2Y and 3M:**

Yahoo Finance ticker coverage for Treasuries:
- `^IRX` = 13-week (3-month) T-bill rate (Yahoo Finance)
- `^FVX` = 5-year
- `^TNX` = 10-year
- `^TYX` = 30-year

The 2-year yield is NOT available as a reliable Yahoo Finance ticker. FRED series `DGS2`
(Market Yield on U.S. Treasury Securities at 2-Year Constant Maturity) is the authoritative
daily series and is used as the primary source for US_2Y_YIELD. The 3-month yield for spread
computation uses yfinance `^IRX` (13-week T-bill rate as a proxy for 3M). The 10Y-3M spread
will use FRED DGS3MO or yfinance ^IRX — the fetch module must verify ^IRX availability and fall
back to FRED DGS3MO if needed.

**Updated yields assignments:**

| Indicator | Indicator ID | Source | Ticker / Series ID |
|-----------|-------------|--------|-------------------|
| US 2Y Treasury Yield | US_2Y_YIELD | FRED | DGS2 |
| US 3M Treasury Yield (for spread) | US_3M_YIELD | yfinance (^IRX) / FRED DGS3MO fallback | ^IRX / DGS3MO |
| US 5Y Treasury Yield | US_5Y_YIELD | yfinance | ^FVX |
| US 10Y Treasury Yield | US_10Y_YIELD | yfinance | ^TNX |
| US 30Y Treasury Yield | US_30Y_YIELD | yfinance | ^TYX |

## Decision Point 3: GDPNow Fetch Method

### Context

GDPNow is the Atlanta Federal Reserve's real-time GDP nowcast for the current quarter. It is
explicitly NOT available via the FRED API (the Atlanta Fed is a separate institution from the
St. Louis Fed). The story requires fetching from the Atlanta Fed website directly (AC3, AC7).

### Evaluation Criteria

| # | Criterion | Weight |
|---|-----------|--------|
| C1 | Returns machine-readable data (not HTML parsing) | must-have |
| C2 | Provides the current quarter's real-time estimate as a single value | must-have |
| C3 | Works with `requests` library (no browser automation) | must-have |
| C4 | Stable URL — low risk of breaking on Atlanta Fed site updates | important |

### Approaches Identified

#### Approach A — Atlanta Fed GDPNow CSV Download Endpoint

**Mechanism:** The Atlanta Fed publishes a CSV file of GDPNow historical estimates at:
```
https://www.atlantafed.org/-/media/documents/cqer/researchcq/gdpnow/GDPNow-model-output.xlsx
```
or equivalently, the downloadable data page at:
```
https://www.atlantafed.org/cqer/research/gdpnow
```

The Atlanta Fed also maintains a direct data download link:
```
https://www.atlantafed.org/-/media/documents/cqer/researchcq/gdpnow/gdpnow-tracking.xlsx
```

As of April 2026, the authoritative machine-readable endpoint for GDPNow is an Excel file
(.xlsx). The fetch module must use `requests` to download the binary, parse with `pandas.read_excel()`,
and extract the most recent estimate. The column containing the GDPNow estimate is typically
labeled "GDPNow" in the first sheet.

**Evidence:** The Atlanta Fed's GDPNow page documents this download. The Excel file is updated
each release day (typically 2-3 times per week during the quarter). It contains date-stamped
estimates allowing extraction of the latest value.

**Limitations:** Excel parsing requires `openpyxl` as an additional dependency (for
`pandas.read_excel()`). URL is subject to Atlanta Fed site maintenance. Binary download is
slightly heavier than a plain CSV but well within practical limits.

#### Approach B — Atlanta Fed Plain Text / JSON (if available)

**Mechanism:** Some Atlanta Fed data feeds are available as plain text or JSON. If a JSON API
or plain-text CSV endpoint exists for GDPNow, it is preferable.

**Evidence:** As of 2026, the Atlanta Fed does not publish GDPNow data via a structured JSON
API. The primary machine-readable format is the Excel spreadsheet. A plain-text CSV version
may exist at:
```
https://www.atlantafed.org/-/media/documents/cqer/researchcq/gdpnow/GDPNow-model-output.csv
```
This URL pattern exists for some Atlanta Fed data series and is worth attempting first. If it
returns a valid CSV, it eliminates the `openpyxl` dependency.

**Evidence update:** The Atlanta Fed GDPNow tracking data is available as a downloadable file.
Confirmed approach: attempt CSV download first; if that returns a non-CSV MIME type or 404,
fall back to the Excel endpoint. This preserves the ability to avoid an extra dependency.

### Decision: Approach A (Excel) with CSV-first attempt

**Primary endpoint to attempt:** CSV at
`https://www.atlantafed.org/-/media/documents/cqer/researchcq/gdpnow/GDPNow-model-output.xlsx`

**Implementation approach:**
1. Download with `requests.get()` using a browser-like User-Agent header (Atlanta Fed may
   block default Python user agents)
2. Parse with `pandas.read_excel()` (requires `openpyxl` — add to requirements.txt)
3. Extract the most recent GDPNow estimate from the last row with a non-null value
4. Store as a single-observation record: `(GDPNOW, <date_of_estimate>, <estimate_value>)`

**AC7 compliance:** If the download fails (network error, HTTP error, parse error), surface a
visible error for GDPNOW only. Do not block other indicators. The error message must identify
the source URL and the failure reason.

**Add to requirements.txt:** `openpyxl>=3.1.0` (required for `pandas.read_excel()` with .xlsx files)

## Decision Point 4: macro_indicators Table Schema

### Context

The new `macro_indicators` table must:
- Store records for all 32+ indicators (plus 2 intermediate spread inputs)
- Support AC8: include indicator_id, date, value, fetch_timestamp
- Support AC9: consistent named indicator_ids for direct z-score computation in E03S02
- Not modify the existing `indicators` table

### Schema Design

```sql
CREATE TABLE IF NOT EXISTS macro_indicators (
    indicator_id    TEXT NOT NULL,
    date            TEXT NOT NULL,
    value           REAL NOT NULL,
    fetch_timestamp TEXT NOT NULL,
    PRIMARY KEY (indicator_id, date)
);
```

**Design decisions:**

1. **Composite primary key `(indicator_id, date)`** — ensures AC5: INSERT OR IGNORE on
   re-fetch will skip records that already exist for that indicator on that date. No overwrites
   or deletions occur.

2. **indicator_id as TEXT** — uses the canonical string identifiers defined in DP2 (e.g.,
   `US_10Y_YIELD`, `CPI_YOY`). Consistent IDs satisfy AC9.

3. **date as TEXT in ISO-8601 (YYYY-MM-DD)** — consistent with the existing `indicators` table
   convention. Monthly indicators use the last day of the reference month as the date (e.g.,
   `2024-03-31` for March 2024 CPI), which matches FRED's convention for monthly series.

4. **value as REAL** — single numeric value per record. YoY % values, yield %, index levels,
   and futures prices all map to REAL. No unit normalization is applied — values are stored
   in their natural units as returned by the source.

5. **fetch_timestamp as TEXT** — ISO-8601 UTC timestamp of when the record was fetched
   (e.g., `2026-04-23T14:30:00Z`). Satisfies AC8.

6. **No source column** — source documentation lives in the fetch module comments and this
   evaluation artefact. Storing source in the DB adds overhead with no downstream benefit for
   E03S02 z-score computation.

7. **No NOT NULL on value** — debated. Decision: value IS NOT NULL. If a fetch returns a null
   value (e.g., FRED returns `.` for missing observations), that row is skipped and not stored.
   A gap in the series is preferable to a NULL stored value that would break z-score computation.

**Index recommendation:**
```sql
CREATE INDEX IF NOT EXISTS idx_macro_indicator_date
    ON macro_indicators (indicator_id, date DESC);
```

This index supports the primary E03S02 access pattern: `SELECT date, value FROM macro_indicators
WHERE indicator_id = ? ORDER BY date DESC LIMIT 52` for rolling window computation.

**Why not a wide table (one column per indicator)?** The normalized design (one row per
indicator per date) is more maintainable when indicator count changes and directly supports
the E03S02 query pattern. The existing `indicators` table uses a wide design appropriate for
its fixed, small column set — `macro_indicators` with 32+ indicators would be unwieldy as
wide format.

3. **Atlanta Fed GDPNow URL:** The Excel endpoint URL must be verified at implementation time.
   The Atlanta Fed periodically reorganizes its web infrastructure. If the Excel URL returns a
   non-200 status, check the Atlanta Fed GDPNow page for the current download link and document
   it in the fetch module.

4. **USD/CNH availability:** CNH=X via yfinance should be confirmed at implementation time.
   Fall back to CNY=X with a code comment if CNH=X is unavailable or returns insufficient
   history.

5. **openpyxl dependency:** Add `openpyxl>=3.1.0` to requirements.txt for GDPNow Excel parsing.
   If the Atlanta Fed CSV endpoint is confirmed available (non-Excel), this dependency can be
   omitted.
