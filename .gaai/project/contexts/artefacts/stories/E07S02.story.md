# Story: Implement SQLite Job Queue for Resumable Delivery State

## User Story

**As a** developer running unattended delivery on Windows,
**I want** delivery job state persisted in a SQLite database (`data/gaai-jobs.db`),
**so that** usage-limit hits or process kills do not permanently orphan `in_progress` stories.

---

## Context

When a usage-limit is hit mid-delivery, `claude -p` exits non-zero and the story's backlog status remains `in_progress` indefinitely. The existing `GAAI_STALENESS_THRESHOLD` only rescues orphans after `delivery_timeout + 10min` — but it relies on the daemon still running, which is not guaranteed on Windows. A SQLite job queue provides durable state that survives process death: on next startup, `gaai_deliver.py` reads the queue, detects orphaned jobs, and resets them to `refined` so delivery can retry.

This story also produces the required DEC for `gaai-jobs.db` storage (separate from DEC-2026-04-22-01 which covers dashboard data only).

---

## Acceptance Criteria

- [ ] AC1: `data/gaai-jobs.db` is created on first run with schema: `jobs(id TEXT PRIMARY KEY, story_id TEXT NOT NULL, status TEXT NOT NULL, started_at TEXT, updated_at TEXT, pid INTEGER, log_path TEXT)`. SQLite WAL mode is enabled on creation.
- [ ] AC2: When `gaai_deliver.py` launches a `claude -p` subprocess for a story, it calls `subprocess.Popen()` to start the process, then immediately inserts a row with `status='running'` and the subprocess PID (from `proc.pid`) into `gaai-jobs.db`. The PID is recorded after `Popen()` returns — the OS assigns the PID only after the process starts.
- [ ] AC3: When a `claude -p` subprocess exits with code 0, the corresponding job row is updated to `status='done'` and `updated_at` is refreshed.
- [ ] AC4: When a `claude -p` subprocess exits with a non-zero code, the corresponding job row is updated to `status='failed'`.
- [ ] AC5: `gaai_deliver.py` exposes a function `_set_backlog_status(story_id: str, new_status: str) -> None` that updates the `status` field for the matching story in `active.backlog.yaml`. Full contract: (a) only the `status:` line for the matching story is modified — all comments, field ordering, and unrelated content must be preserved; the replacement must be scoped to the matching story block: locate the story by finding its `id: {story_id}` field first, then replace only the `status:` line within that block — use targeted regex, not YAML round-trip serialization, to avoid corrupting adjacent story fields; (b) if `story_id` is not found in the file, raises `ValueError("story_id not found in active.backlog.yaml: {story_id}")`; (c) writes atomically: write to `active.backlog.yaml.tmp` in the same directory, then `os.replace()` to `active.backlog.yaml` — the `.tmp` write and `os.replace()` are wrapped in a try/finally: if any exception occurs after `.tmp` is created but before `os.replace()` completes, the finally block must attempt to delete `active.backlog.yaml.tmp`, catching and silently ignoring `FileNotFoundError` (to handle the case where the file was never created before the exception) — any other exception from the deletion is also silently ignored in the finally block so the original exception is not masked (same-directory temp file ensures the rename is intra-volume and never raises on Windows); (d) `filelock.FileLock("active.backlog.yaml.lock", timeout=30)` in the same directory as `active.backlog.yaml` is acquired and held across the full read-modify-write cycle before releasing; if the lock cannot be acquired within 30 seconds, `filelock.Timeout` is raised — the caller must catch this and log "ERROR: could not acquire backlog lock for {story_id} after 30s" before propagating the error. The 30-second timeout prevents indefinite deadlock if a crashed process holds the lock file on Windows.
- [ ] AC6: On startup, `gaai_deliver.py` queries `gaai-jobs.db` for rows with `status='running'`. For each such row, it checks if the stored PID is an active process using `psutil.pid_exists(pid)` (required over `os.kill(pid, 0)` due to PID recycling on Windows). If the PID is not active: (a) the backlog entry is reset to `status: refined` via `_set_backlog_status(story_id, 'refined')`, authorized by DEC-2026-05-12-02; (b) the job row is updated to `status='orphaned'`. Both updates are logged to `data/gaai-deliver.log`. Note: `_set_backlog_status()` is cross-platform (pure Python, no shell dependency) — the Notes section's framing as "Windows-only" refers to its role as a Windows-native *replacement* for `backlog-scheduler.sh`; the function itself is callable from any platform where `gaai_deliver.py` runs.
- [ ] AC7 (error-handling): SQLite concurrent writes — two `claude -p` subprocesses updating their own job rows simultaneously — are handled without data loss. SQLite WAL mode allows concurrent reads without blocking writes, but concurrent writers still contend for the write lock and may receive `SQLITE_BUSY`. Retry with exponential backoff is implemented for write operations (`INSERT` and `UPDATE` statements on `gaai-jobs.db`) that raise `sqlite3.OperationalError` with message text containing "database is locked". Retry contract: 3 total attempts (attempt 0 = initial try at 100ms delay if it fails, attempt 1 = 200ms delay, attempt 2 = 400ms delay) — "max 3 retries" means 3 total attempts, not 3 additional retries after a first failure; backoff formula is `100ms * 2^attempt`; no jitter. If all 3 retries are exhausted: the exception is re-raised to the caller (not swallowed); the job row is left in its pre-write state (no partial update is committed). The retry wraps each complete `cursor.execute()` + `conn.commit()` call — if `conn.commit()` raises, the retry re-executes the full `cursor.execute()` + `conn.commit()` pair to avoid stale-state partial retries.
- [ ] AC8: Pre-condition — `DEC-2026-05-12-01` must exist as a section in `.gaai/project/contexts/memory/decisions/_log.md` before this story is marked done; verify by reading the file and asserting the DEC ID is present. Actionable deliverable — `data/gaai-jobs.db` must be listed in `.gitignore`; if the entry is absent, add it (idempotent: check before adding to avoid duplicates). Note: DEC-2026-05-12-02's Impact field incorrectly references "E07S02 AC5" — the authorized recovery call (`_set_backlog_status(story_id, 'refined')`) is in AC6, not AC5 (AC5 defines the function; AC6 calls it). This mislabel is a known audit trail issue in the DEC log; it does not affect governance validity.

## Notes

- `data/gaai-jobs.db` is listed in `.gitignore` — local state, not committed to the repo
- The `jobs` table uses `story_id` (e.g., `E07S03`) as the natural key for lookups; `id` is a UUID generated at insert time
- Two separate concurrency mechanisms serve two different files: `filelock` serializes concurrent calls to `_set_backlog_status()` writing `active.backlog.yaml` (YAML is not atomically writable on Windows); SQLite WAL + retry handles concurrent writes to `gaai-jobs.db`
- `backlog-scheduler.sh` is the authoritative tool for backlog status updates on Linux/macOS. On Windows (where bash is unavailable), `_set_backlog_status()` is the Python equivalent — it honors the same YAML format conventions but is implemented natively
- `psutil` and `filelock` must be added to `requirements.txt`
