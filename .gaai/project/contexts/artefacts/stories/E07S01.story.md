# Story: Build Python Delivery Entrypoint with Platform Detection

## User Story

**As a** developer on Windows,
**I want** a Python entrypoint (`gaai_deliver.py`) that detects my OS and either launches the native bash daemon on Linux/macOS or runs the Windows delivery loop directly,
**so that** I can start delivery without the bash scripts unconditionally aborting on Windows.

---

## Context

`daemon-start.sh:35-40` and `delivery-daemon.sh:126-135` both contain a `case` block that matches MINGW/MSYS/CYGWIN and unconditionally exits with code 1. There is no fallback â€” even WSL fails due to PATH persistence issues. This story creates `gaai_deliver.py` as the new primary entrypoint: it detects the current platform via `sys.platform` and routes to the appropriate execution path. Linux/macOS behavior is unchanged.

---

## Acceptance Criteria

- [ ] AC1: Given Windows (`sys.platform == 'win32'`), when `gaai_deliver.py` is invoked with no mode flags, then it: (a) reads `active.backlog.yaml` and identifies stories with `status: refined` â€” if the file is missing or contains invalid YAML, exits 1 with "ERROR: cannot read active.backlog.yaml: {reason}"; (b) for each refined story, launches one `claude -p` subprocess using `CLAUDE_SUBPROCESS_ARGS` plus the story context as input; (c) waits for each subprocess to exit before launching the next; (d) exits 0 after all subprocesses complete, or 1 on the first subprocess failure â€” "subprocess failure" is defined as `proc.returncode != 0`. Does NOT invoke `daemon-start.sh` or `delivery-daemon.sh`.
- [ ] AC2: Given Linux or macOS (`sys.platform in ('linux', 'darwin')` or `sys.platform.startswith('linux')`), when `gaai_deliver.py` is invoked, then it delegates to `daemon-start.sh` â€” resolved as `Path(__file__).absolute().parent / '.gaai' / 'core' / 'scripts' / 'daemon-start.sh'` (`.absolute()` anchors `__file__` to the filesystem before traversal, avoiding CWD-dependency when `__file__` is a relative path) â€” with all arguments passed through unchanged. No behavioral change on non-Windows platforms.
- [ ] AC3: `gaai_deliver.py` defines a module-level list constant `CLAUDE_SUBPROCESS_ARGS` containing at minimum `['--dangerously-skip-permissions']`. This constant holds all fixed `claude -p` CLI flags; per-invocation arguments (the `-p` flag and story input) are combined with this constant at call time, not stored in it.
- [ ] AC4: Given any platform, when `gaai_deliver.py` exits cleanly, then exit code is 0; when it exits on error, then exit code is 1. Given a platform where `sys.platform` does not match `win32`, `linux`, `darwin`, or any string starting with `linux`, `gaai_deliver.py` exits 1 with "ERROR: unsupported platform: {sys.platform}" printed to stderr.
- [ ] AC5 (error-handling): Given Linux/macOS and `daemon-start.sh` is not found at its resolved path, when `gaai_deliver.py` is invoked, then it exits 1 and prints to stderr: "ERROR: daemon-start.sh not found at {resolved_path}" â€” no silent failure.
- [ ] AC6 (error-handling): `gaai_deliver.py` uses `pathlib.Path` for all internal file path construction â€” no string concatenation, `os.path.join`, or `os.getcwd()` for path operations. This is verifiable by code inspection: zero string-based path constructions in the source.
- [ ] AC7: All `print()` and `sys.stderr.write()` output from `gaai_deliver.py` is ASCII-only â€” no Unicode characters that crash the Windows console.
