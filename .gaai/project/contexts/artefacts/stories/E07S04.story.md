# Story: Add --unattended Mode and Task Scheduler Integration

## User Story

**As a** developer on Windows,
**I want** `gaai_deliver.py --unattended` to run safely under Windows Task Scheduler without human supervision,
**so that** stories are delivered overnight without interactive input or manual intervention.

---

## Context

The bash daemon uses `tmux` (or `Terminal.app` on macOS) to launch background sessions. Neither is available on native Windows. The Windows architecture is Task Scheduler per-poll: Windows Task Scheduler fires `gaai_deliver.py` on a schedule; each invocation reads the backlog, checks active jobs in `gaai-jobs.db`, launches `claude -p` subprocesses for ready stories within the concurrency limit, and exits. Re-entrant safety (what happens if Task Scheduler fires while a prior invocation is running) is enforced by the Task Scheduler "do not start a new instance" setting — not by application-level locking.

---

## Acceptance Criteria

- [ ] AC1: `gaai_deliver.py --unattended` runs `--doctor` before any delivery work, routing all `--doctor` output to `data/gaai-deliver.log` (created on first run if absent; appended thereafter; `data/gaai-deliver.log` must be listed in `.gitignore` — this entry is already present from project setup; Delivery must verify it exists, not add a duplicate). If `--doctor` exits 1 (any ERROR check), the unattended run aborts immediately with exit 1 and appends "ABORTED: doctor reported N error(s) - resolve before retrying" (ASCII hyphen, not em-dash) to `data/gaai-deliver.log`. N is extracted by parsing the `--doctor` summary line (`doctor: N checks, N fixed, N warnings, N errors`) for the error count. No delivery is attempted in a broken environment.
- [ ] AC2: `gaai_deliver.py --unattended` does not present any interactive prompts — all confirmations, y/n inputs, and user prompts are suppressed or bypassed. Verified by running `gaai_deliver.py --unattended` with stdin redirected to NUL (`python gaai_deliver.py --unattended < NUL` on Windows) and asserting the process exits within 60 seconds with no hang — any blocking read on stdin would cause the process to stall indefinitely, making a successful exit within 60 seconds the test vector. (This test vector verifies the Python process itself does not block on stdin; it does not test subprocess-level prompts, which are governed by `--dangerously-skip-permissions` in AC3.)
- [ ] AC3: All `claude -p` subprocess invocations include `--dangerously-skip-permissions` as a CLI flag (via `CLAUDE_SUBPROCESS_ARGS` from E07S01 AC3).
- [ ] AC4: A `gaai-deliver.bat` wrapper is committed to the project root. Its content must be equivalent to:
  ```bat
  @echo off
  setlocal enabledelayedexpansion
  if not defined PYTHON_EXE (
      where python >nul 2>&1 || (
          echo ERROR: python not found. Set PYTHON_EXE to the full Python path. See docs/windows-task-scheduler.md
          exit /b 1
      )
      set PYTHON_EXE=python
  )
  cd /d "%~dp0"
  "!PYTHON_EXE!" "%~dp0gaai_deliver.py" --unattended %*
  ```
  `setlocal enabledelayedexpansion` is required because `PYTHON_EXE` may be set inside the parenthesized `if`-block; delayed expansion (`!PYTHON_EXE!` on the invocation line) ensures the value assigned inside the block is visible to subsequent commands. `%~dp0` always includes a trailing backslash and is evaluated by cmd.exe at parse time — brackets in the path are literal characters and require no special escaping. `%*` passes through any extra arguments unchanged. The `PYTHON_EXE` guard prevents silent failure when PATH is stripped by Task Scheduler. `gaai-deliver.bat` is committed to the repo; it is NOT listed in `.gitignore`.
- [ ] AC5: `gaai-deliver.bat` sets the working directory to the project root via `cd /d "%~dp0"` before invoking Python, so relative paths in `gaai_deliver.py` resolve correctly regardless of Task Scheduler's default working directory.
- [ ] AC6: A `docs/windows-task-scheduler.md` file is created by this story. It must contain all five of: (a) the phrase `gaai-deliver.bat` with instruction to use its full path as the "Start a program" action; (b) the exact substring `PYTHON_EXE` with instruction to set it to the full Python executable path in Task Scheduler's environment; (c) the exact substring `working directory` with instruction to set it to the project root; (d) the exact phrase "do not start a new instance" as the mandatory overlap policy setting; (e) the exact substring `orphaned` in the context of the recycled-PID manual recovery procedure (documenting the `UPDATE jobs SET status='orphaned'` escape path described in AC9). Verified by asserting the file exists and contains each of these five exact substrings: `gaai-deliver.bat`, `PYTHON_EXE`, `working directory`, `do not start a new instance`, and `orphaned`.
- [ ] AC7 (error-handling): Given `--unattended` is invoked and `gaai-jobs.db` does not exist (first run), then `gaai-jobs.db` is created automatically (E07S02 AC1 init) before any delivery attempt — no crash on missing DB.
- [ ] AC8 (error-handling): Given `--unattended` and the active backlog contains no `refined` stories, when `gaai_deliver.py` runs, then it exits 0 cleanly with log entry "nothing to deliver" in `data/gaai-deliver.log` — no error, no hang.
- [ ] AC9 (error-handling): Given `--unattended` and `gaai-jobs.db` contains a row with `status='running'` whose stored PID is active (`psutil.pid_exists(pid)` is True), when `gaai_deliver.py` runs, then that story is NOT re-launched and its job row status is NOT modified — it is counted against the concurrency limit and skipped. Note: PID recycling is a known limitation — if a different process has inherited the stored PID, `psutil.pid_exists(pid)` returns True and the story slot is incorrectly counted as occupied. If that recycled PID belongs to a long-lived process (e.g., a Windows system service), the story slot may be permanently occupied and will never self-clear through the normal orphan detection cycle (which only detects dead PIDs). In this case, `--doctor` staleness detection (E07S03 AC5) will also not catch it, because the backlog entry remains `in_progress` with a recent-enough `started_at`. **Escape path**: operator must manually update or delete the stuck row in `gaai-jobs.db` (`UPDATE jobs SET status='orphaned' WHERE story_id='...'`) and then reset the backlog entry to `refined` via `_set_backlog_status()`. This manual intervention path is documented in `docs/windows-task-scheduler.md`. This edge case is explicitly acknowledged as out of scope for automatic recovery in this story.

## Notes

- If Task Scheduler single-instance setting is NOT configured and two `gaai-deliver.bat` invocations overlap: the SQLite job queue (E07S02) prevents double-launch of the same story (a story already `status='running'` in `gaai-jobs.db` is skipped per AC9). No data corruption occurs, but the concurrency limit may be exceeded.
- `gaai-deliver.bat` is committed to the repo root as a template. The `%PYTHON_EXE%` override mechanism allows machine-specific Python paths without committing user-specific absolute paths to the repo.
- `data/gaai-deliver.log` is listed in `.gitignore`. `gaai-deliver.bat` is committed to the repo (not ignored) — machine-specific Python path is handled by the `%PYTHON_EXE%` environment variable, not by baking it into the file.
