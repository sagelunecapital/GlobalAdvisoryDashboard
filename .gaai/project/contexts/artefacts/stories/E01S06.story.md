# Story E01S06: Initialize and Maintain Historical Indicator Database

## User Story

**As a** fund manager,
**I want** the dashboard to maintain a persistent database of historical indicator values going back at least one year,
**so that** divergence detection has reliable historical data to compare against, and all data is preserved for future analysis.

---

## Context

Divergence detection (E01S02) requires comparing current SPX and MMTH values against prior swing highs and lows. A minimum of one year of historical data ensures meaningful swing comparisons are always available. This database is append-only — data is never deleted. New daily values are added to it on each scheduled refresh.

The four indicators to store: SPX price, SPX 12-day EMA, SPX 25-day EMA, MMTH.

---

## Acceptance Criteria

- [ ] AC1: On initial setup, the system loads at least 1 year of daily historical data for all four indicators (SPX price, SPX 12d EMA, SPX 25d EMA, MMTH) into a persistent database
- [ ] AC2: The database schema stores date, SPX daily high price, SPX 12d EMA, SPX 25d EMA, and MMTH value per record
- [ ] AC3: The database is append-only — no existing records are modified or deleted under any condition
- [ ] AC4: Each daily refresh (manual or scheduled) appends the new day's indicator values to the database
- [ ] AC5: The database persists across application restarts — data is not held in memory only
- [ ] AC6 (error-handling): If the initial historical data load fails partway through, the system surfaces an error and leaves a clearly retryable state — partial progress is either persisted with a clear resumption point, or cleanly rolled back so the next attempt starts fresh; in no case is partial data silently mixed with a complete load in a way that could make an incomplete database appear complete

---

## Out of Scope

- Data visualisation or charting of historical values
- User access to query or export the database directly
- Data for indicators beyond the four defined in S-1

---

## Dependencies

None — this is the foundation story; all other stories that need historical data depend on this one.

---

## Notes

Delivery chooses the database technology (SQLite, PostgreSQL, flat-file store, etc.) — all open; no technology constraint has been declared. The append-only constraint (AC3) is a business requirement, not a technical default. The initial historical load source may differ from the ongoing daily fetch source — Delivery decides.
