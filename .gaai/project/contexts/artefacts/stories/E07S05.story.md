# Story: Integration Tests for All Documented Failure Modes

## User Story

**As a** developer maintaining the GAAI daemon,
**I want** integration tests that simulate each documented Windows failure mode,
**so that** regressions are caught before they surface in unattended delivery sessions.

---

## Context

Each of the prior E07 stories addresses a specific failure mode observed across real delivery sessions. This story encodes those failure modes as repeatable, automated tests. Tests use mocked subprocesses and file system state ‚Äî no live `claude -p` invocations, no live Task Scheduler interactions. All tests must pass under `pytest` with no dependencies beyond `requirements.txt` plus `filelock` and `psutil` (both added in E07S02).

Failure mode taxonomy (S-4):
- S-4(a-i): bash scripts hard-exit on Windows (daemon-start.sh / delivery-daemon.sh)
- S-4(a-ii): tmux unavailable on Windows (Python delivery path must not invoke tmux)
- S-4(b): stale PID in gaai-jobs.db produces orphaned in_progress story
- S-4(c): missing `skipDangerousModePermissionPrompt` in `~/.claude/settings.json`
- S-4(d): `claude` binary not in PATH
- S-4(e): concurrent SQLite writes to `gaai-jobs.db` (structural robustness)

---

## Acceptance Criteria

- [ ] AC1: Test simulates S-4(a-i) (bash hard-exit guard): given mocked `sys.platform == 'win32'`, when `gaai_deliver.py` runs, then it does NOT invoke `daemon-start.sh` or `delivery-daemon.sh` ‚Äî verified by asserting no subprocess call to any `*.sh` file occurs.
- [ ] AC2: Test simulates S-4(a-ii) (tmux unavailable on Windows): given Windows platform, when `gaai_deliver.py` runs, then it proceeds without invoking `tmux` ‚Äî no `tmux`-not-found exception or exit 1. (The Python delivery path never calls tmux; this test verifies the guard is complete.)
- [ ] AC3: Test simulates S-4(b) (orphaned in_progress story via stale PID): given `gaai-jobs.db` (created in `tmp_path`) containing a row with `status='running'` and a PID that `psutil.pid_exists()` reports as inactive ‚Äî mocked via `monkeypatch.setattr('gaai_deliver.psutil.pid_exists', lambda pid: False)` (assumes `gaai_deliver.py` uses `import psutil` and calls `psutil.pid_exists()`; patch the reference in the module namespace, not the source module) ‚Äî when `gaai_deliver.py` starts, then (a) the job row is updated to `status='orphaned'`, and (b) `_set_backlog_status(story_id, 'refined')` is called ‚Äî verified by mocking `gaai_deliver._set_backlog_status` and asserting it was called with the correct `story_id` and `'refined'`.
- [ ] AC4: Test simulates S-4(c) (missing skipDangerousModePermissionPrompt): the `--doctor` settings path constant in `gaai_deliver.py` is named `CLAUDE_SETTINGS_PATH`. The E07S03 implementation MUST read this constant by module-global name at call time ‚Äî not capture it in a closure or default argument at import time ‚Äî so that monkeypatching it after import takes effect. The test monkeypatches `gaai_deliver.CLAUDE_SETTINGS_PATH` to point to `tmp_path / 'settings.json'`. Given that file does not contain `skipDangerousModePermissionPrompt`, when `--doctor` runs, then the key is added to the `tmp_path` file and `--doctor` exits 0. The real `Path.home() / '.claude' / 'settings.json'` is never modified.
- [ ] AC5: Test simulates S-4(d) (claude binary not in PATH): given `shutil.which('claude')` returns `None` (mocked via `monkeypatch.setattr('shutil.which', lambda name: None)`), when `gaai_deliver.py` attempts to launch a story, then it exits 1 and prints to stderr "ERROR: claude not found in PATH" ‚Äî no `FileNotFoundError` traceback propagates.
- [ ] AC6: Test simulates S-4(e) (concurrent SQLite writes): two threads simultaneously call the job-status update function for different story IDs against the same `gaai-jobs.db` (created in `tmp_path`); the test asserts (a) both rows are present with correct final status after both threads complete, and (b) no `SQLITE_BUSY` exception propagates (retry logic absorbs it).
- [ ] AC7 (error-handling): All tests mock `subprocess.Popen` and `subprocess.run` ‚Äî no live `claude -p` or `bash` invocations occur during the test suite. Test fixtures restore all modified state via `tmp_path` or teardown: `settings.json` modifications use the `tmp_path`-patched constant (AC4); `gaai-jobs.db` is created in `tmp_path` (AC3, AC6); `.delivery-locks/` modifications use a `tmp_path` directory.
- [ ] AC8: All seven tests (AC1‚ÄìAC6 plus AC9) pass under `pytest` with exit code 0. `filelock` and `psutil` are the only dependencies added beyond the existing `requirements.txt`.
- [ ] AC9 (error-handling): A test verifies the `--doctor` exit code contract by mocking individual check functions (e.g., `monkeypatch.setattr('gaai_deliver._check_python', lambda: ('ERROR', 'python not found'))`). Given one mocked check returning ERROR and two returning OK, `--doctor` exits 1. Given all mocked checks returning OK or Fixed, `--doctor` exits 0. WARN checks do not affect the exit code. The mock mechanism must patch the individual check function names used inside `gaai_deliver.py` ‚Äî Delivery must expose each check as a named callable (e.g., `_check_python`, `_check_settings_path`, etc.) so they are individually mockable.
