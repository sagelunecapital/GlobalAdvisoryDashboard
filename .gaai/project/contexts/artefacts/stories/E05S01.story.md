# Story E05S01: Fetch and Persist SOFR, IORB, EURCHF, and GDP YoY%

## User Story

**As a** fund manager,
**I want** the dashboard to automatically retrieve and store the four data series required for the Regime section that are not already covered by existing modules,
**so that** the Regime signal computation has a complete, persistent data foundation.

---

## Context

The Regime section (E05) synthesises signals from existing DB tables (E02 and E03) plus four series that no existing epic fetches: SOFR (Secured Overnight Financing Rate), IORB (Interest on Reserve Balances), EURCHF 5-day performance, and GDP YoY% (quarterly GDP Percent Change from Year Ago). This story establishes the DB tables for those four series. SOFR and IORB come from the FRED API with weekly frequency, ending-Friday, average aggregation. EURCHF comes from the barchart performance page (`https://www.barchart.com/forex/quotes/%5EEURCHF/performance`). GDP YoY% comes from the FRED `GDP` series with `units=pc1` (Percent Change from Year Ago, quarterly).

---

## Acceptance Criteria

- [ ] AC1: SOFR is fetched from the FRED API (series `SOFR` with weekly frequency, ending-Friday, average aggregation) and persisted with fields `(indicator_id, date, value, fetch_timestamp)`. At least 52 weekly observations are loaded on initial run.
- [ ] AC2: IORB is fetched from the FRED API (series `IORB` with weekly frequency, ending-Friday, average aggregation) and persisted with the same schema as AC1. At least 52 weekly observations loaded on initial run.
- [ ] AC3: EURCHF 5-day performance is fetched from `https://www.barchart.com/forex/quotes/%5EEURCHF/performance` and stored as a weekly observation with fields `(indicator_id, date, value, fetch_timestamp)`. Value represents the 5-day percentage change.
- [ ] AC4: GDP YoY% is fetched from the FRED API (`GDP` series, `units=pc1`) and persisted with fields `(indicator_id, date, value, fetch_timestamp)`. At least 8 quarterly observations (2 years) are loaded on initial run.
- [ ] AC5 (data-freshness): Given a subsequent refresh run, only observations with dates not already present in the DB are fetched and appended — existing records are never overwritten or deleted. For `EURCHF_5D_PERF`, the date is normalized to the Friday of the week in which the scrape is performed, ensuring non-overlapping weekly records.
- [ ] AC6 (error-handling): Given any single indicator fetch fails (network error, API limit, parse error), the system surfaces a visible error identifying the failed indicator by name and continues fetching the remaining indicators — partial success is stored, not discarded.
- [ ] AC7 (error-handling): Given the FRED API key is absent or invalid, the system raises a clear error naming the missing credential — it does not silently return empty data.
- [ ] AC8 (signal-detection): Each indicator is stored under the exact `indicator_id` values listed below — these are the canonical IDs that E05S02 queries by name with no transformation:
  - `SOFR_WEEKLY` (SOFR weekly average)
  - `IORB_WEEKLY` (IORB weekly average)
  - `EURCHF_5D_PERF` (EURCHF 5-day performance)
  - `GDP_YOY_PCT` (GDP Percent Change from Year Ago)

---

## Out of Scope

- Computing any derived signals from these series — handled in E05S02
- Displaying data — handled in E05S03
- Fetching any series already in scope for E02 or E03 (Net Liquidity, T5YIE, US02Y, US10Y, DXY, USDJPY, USDCNH, GDPNow)
