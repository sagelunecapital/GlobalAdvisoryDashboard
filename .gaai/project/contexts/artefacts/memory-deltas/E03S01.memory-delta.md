### FRED (Federal Reserve Economic Data)

- **Source:** St. Louis Fed REST API — `https://api.stlouisfed.org/fred/series/observations`
- **Auth:** `FRED_API_KEY` environment variable (free registration at fred.stlouisfed.org)
- **Series used:**
  - Inflation: `CPIAUCSL` (CPI), `CPILFESL` (Core CPI), `PCEPI` (PCE), `PCEPILFE` (Core PCE), `PPIACO` (PPI)
  - Yields: `DGS2` (US 2Y), `DGS3MO` (3M, intermediate for spread), `EFFR` (effective Fed funds)
  - Breakeven: `T5YIE` (5Y breakeven), `T2YIE` (2Y breakeven)
  - Growth: `NAPM` (ISM Mfg PMI, legacy series), `NMFCI` (ISM Services composite), `UNRATE`, `PAYEMS` (NFP), `RSAFS` (Retail Sales advance)
- **Missing value convention:** FRED uses `"."` for missing observations — filtered out before storage.

### Atlanta Fed GDPNow

- **Source:** `https://www.atlantafed.org/-/media/documents/cqer/researchcq/gdpnow/GDPNow-model-output.xlsx`
- **Data:** Most recent real-time estimate of current quarter GDP growth (annualized %). Single point stored per fetch.
- **Column detection:** Looks for column containing "gdpnow" or "nowcast" (case-insensitive); numeric fallback if single numeric column.
- **Auth:** None required.
- **Dependency:** `openpyxl` required for Excel parsing.

## Delta 3 — DECISIONS-LOG: 3 new decisions

**Target file:** `decisions/_log.md`

### DEC-2026-04-23-01 — Macro DB Layer: Self-Contained Subpackage (No E01 Import)

**Context:** E03S01 needed to add macro indicator storage to the project, which already had `src/db/` (E01 indicators table) and `src/fetch/` (E01 fetch).
**Decision:** Macro data layer (`src/macro/`) is a self-contained subpackage with its own schema, append, and fetch modules. It does NOT import from `src/db/` or `src/fetch/`. The E01 and macro pipelines are fully independent.
**Rationale:** Prevents cross-module coupling and allows each module to evolve independently. The E01 schema is optimized for regime indicators (wide table); the macro schema uses a long/normalized table (indicator_id, date, value) better suited for 32+ varied series.
**Impact:** E03S02–E03S05 read from `macro_indicators` table. No dependency on E01 `indicators` table. Future modules should follow the same self-contained pattern.
**Date:** 2026-04-23

---

### DEC-2026-04-23-02 — FRED REST API Direct (No fredapi Library)

**Context:** FRED provides both a Python library (`fredapi`) and a REST API. E03S01 needed to fetch ~15 FRED series.
**Decision:** FRED data is fetched via direct `requests` calls to `api.stlouisfed.org/fred/series/observations` (JSON). No `fredapi` library dependency.
**Rationale:** The REST API is stable, well-documented, and avoids an additional dependency. `fredapi` offers no material advantage for simple series-fetch use cases. Keeps `requirements.txt` minimal.
**Impact:** All future FRED fetches in this project use the same REST pattern in `src/macro/fetch/fred.py`. `FRED_API_KEY` environment variable is required for production runs.
**Date:** 2026-04-23

---

### DEC-2026-04-23-03 — Macro Indicator Schema: Normalized Long Table

**Context:** E01 uses a wide table (`indicators`: one column per indicator). E03 needs to store 32 indicators across heterogeneous update frequencies.
**Decision:** Macro indicators are stored in a normalized long table: `macro_indicators(indicator_id TEXT, date TEXT, value REAL, fetch_timestamp TEXT)` with composite PK `(indicator_id, date)`.
**Rationale:** A wide table with 32+ columns becomes unmaintainable as the macro module grows. The normalized schema handles indicators with different frequencies (daily vs monthly) cleanly. E03S02 rolling computations need only `SELECT date, value WHERE indicator_id = ?` — no transformation required.
**Impact:** All E03 stories (E03S02–E03S05) query the `macro_indicators` table. The schema is established in `src/macro/db/macro_schema.py`. WAL mode matches E01 DB conventions.
**Date:** 2026-04-23

## Delta 4 — PATTERNS-001 Update: Code Patterns (append)

**Target file:** `patterns/conventions.md`

### Pattern: Per-Indicator Error Isolation (Orchestrator Pattern)

When an orchestrator fetches multiple data series (N indicators), each series fetch and persist operation is wrapped in an independent `try/except` block. Errors are accumulated in a `results` dict (`indicator_id → "ok" | "error: <msg>"`). The orchestrator raises only when ALL indicators fail (`ok_count == 0`). This pattern is implemented in `macro_historical_load.py`.

**When to use:** Any multi-indicator fetch-and-persist orchestrator.
**Key invariant:** A single indicator failure must never block others from being stored.

### Pattern: INSERT OR IGNORE with Composite PK for Append-Only Series

Time series data is persisted using `INSERT OR IGNORE` against a composite primary key `(series_id, date)`. Duplicates are silently skipped; original values are never overwritten. Row count returned is actual inserts (not attempts). This pattern is established in `macro_append.py` (and also in E01's `db/append.py`).

**When to use:** All append operations for time series data in this project.

### Pattern: FRED Series — observation_start + "." Filter

FRED series are fetched from `observation_start="2000-01-01"` to ensure sufficient historical depth. Observations with value `"."` (FRED missing-value sentinel) are filtered out before conversion to float. This ensures only actual released values are stored.

**When to use:** Any FRED series fetch in this project.
