### 2. Patterns & Conventions — conventions.md (PATTERNS-001)

**What exists:** Empty (placeholder sections only).

**New patterns introduced by E01S02:**

| Pattern | Description |
|---------|-------------|
| `_db_rows` injection for testing | Functions that query the database accept an optional `_db_rows` override for test isolation, bypassing the DB entirely. Used in `detect_divergence()`. This is a recurring pattern to adopt for future analysis functions. |
| Swing anchor via global extremum + slice | Bearish: global trough → pre-trough slice max. Bullish: global peak → post-peak slice min. This prevents same-run anchor drift. Pattern should be documented so future divergence variants (e.g., sector-level) follow the same logic. |
| DATA_GAP as distinct unavailable signal | Never conflate DATA_GAP with NO_DIVERGENCE. Functions returning analysis results should use a 4-member enum (or equivalent) that distinguishes "clean no-signal" from "data unavailable." |
| Enum-based result types | Analysis functions return `(ResultEnum, str)` tuples — the enum for programmatic consumption, the string for human-visible explanation. |

**Recommended update to conventions.md:** Add "Code Patterns" entries for all four patterns above.
