## Approach Decisions (from E01S06.approach-evaluation.md)

### DP1 — Programming Language: Python 3.x

**Rationale:** Dominant on all must-have criteria. `yfinance` and `pandas` provide direct access to SPX OHLC (including daily high) and MMTH data. EMA computation is one line via `pandas.ewm()`. Built-in `sqlite3` module eliminates a DB driver dependency. No viable alternative offers comparable library coverage for the specific data sources required.

### DP2 — Database Technology: SQLite

**Rationale:** The only candidate satisfying both zero-ops overhead AND ACID guarantees simultaneously. PostgreSQL fails on ops overhead; CSV fails on ACID (AC6 cannot be satisfied with flat files — no atomic append). SQLite WAL mode provides full ACID with a single-file, no-server design. Built-in Python `sqlite3` module means zero extra dependencies. Append-only constraint (AC3) is enforced at the application layer (no UPDATE/DELETE statements issued; schema and test verify this).

### DP3 — Data Fetch Strategy: yfinance primary; EODData fallback for MMTH

**Rationale:** `yfinance` is the simplest path — single library for both SPX (`^GSPC`) and potentially MMTH (`^MMTH`). SPX daily high is confirmed available and accurate. MMTH via `^MMTH` on Yahoo Finance is not confirmed — the Delivery Agent MUST verify this as the first runtime step. If `^MMTH` returns valid data (1+ year daily history), use yfinance for both. If not, implement the EODData fallback fetch for MMTH. This decision point is resolved at implementation time, not now.

### AC6 Error Handling — Chosen Approach: Option A (Transactional Rollback)

**Rationale over Option B (Resume Checkpoint):** The initial historical load is approximately 260 rows of data. A transactional rollback is simpler, safer, and leaves no ambiguous intermediate state. Re-running the load from scratch after a failure is trivial (seconds) at this data volume. Option B (resume checkpoint) would add ~100 lines of state-management code to solve a problem that doesn't meaningfully exist at this scale. The ACID guarantee from SQLite makes Option A reliable: if the transaction is not committed, the DB is clean and the next run starts fresh automatically.

### Step 3 — Schema Definition

The `indicators` table schema (compliant with DEC-2026-04-18-01):

```sql
CREATE TABLE IF NOT EXISTS indicators (
    date         TEXT PRIMARY KEY,   -- ISO-8601 format: YYYY-MM-DD
    spx_daily_high REAL NOT NULL,    -- SPX daily HIGH price (DEC-2026-04-18-01)
    spx_12d_ema  REAL NOT NULL,      -- 12-day EMA of spx_daily_high
    spx_25d_ema  REAL NOT NULL,      -- 25-day EMA of spx_daily_high
    mmth         REAL NOT NULL       -- % stocks above 200-day MA
);
```

Column naming rules:
- `spx_daily_high` — NOT `spx_price`, NOT `spx_close` (DEC-2026-04-18-01 mandatory)
- `date` as TEXT PRIMARY KEY in ISO-8601 format — SQLite TEXT comparisons on ISO-8601 dates are lexicographically correct

Enable WAL mode on connection creation:
```python
conn.execute("PRAGMA journal_mode=WAL")
```

### Step 4 — SPX Fetch and EMA Computation

Fetch strategy:
- Use `yfinance.download('^GSPC', period='2y', interval='1d')` to fetch 2 years of SPX data
- Extract the `High` column → `spx_daily_high`
- Compute 12-day EMA: `spx_daily_high.ewm(span=12, adjust=False).mean()` → `spx_12d_ema`
- Compute 25-day EMA: `spx_daily_high.ewm(span=25, adjust=False).mean()` → `spx_25d_ema`

**EMA warm-up rationale:** Fetching 2 years (~504 trading days) instead of exactly 1 year ensures that EMA values at the 1-year boundary have 252+ prior data points for warm-up, making the computed EMAs accurate. The final stored dataset is trimmed to the most recent 1 year (252+ trading days) AFTER EMA computation, not before.

Trim logic: after EMA computation on the full 2-year set, filter to `date >= (today - 365 days)` for storage. This ensures EMA accuracy at the oldest stored date.

### Step 5 — MMTH Fetch

If yfinance path: `yfinance.download('^MMTH', period='2y', interval='1d')['Close']` — MMTH is typically a price-like value where `Close` is the relevant column (the % of stocks).

If EODData fallback: parse the downloaded CSV, extract `Date` and `Close` columns, align date format to ISO-8601.

Output contract (both paths): a pandas Series with `DatetimeIndex` and float values, covering at least 252 trading days.

Date alignment: SPX and MMTH dates must be inner-joined before storage — only dates where BOTH data sources have values are stored. This prevents partial rows.

### Step 6 — Historical Load with Transactional Rollback (AC1, AC2, AC3, AC6)

```python
def load_historical(db_path: str) -> None:
    """
    Load 1+ year of historical indicator data into the DB.
    Entire operation is wrapped in a single transaction.
    On any failure: transaction is rolled back, DB is left empty/clean.
    Re-running this function after failure will start fresh.
    """
    conn = get_connection(db_path)
    try:
        conn.execute("BEGIN")
        # 1. Fetch SPX (with 2y warm-up)
        # 2. Compute EMAs
        # 3. Fetch MMTH
        # 4. Inner-join on date
        # 5. Trim to 1-year window
        # 6. INSERT all rows
        conn.execute("COMMIT")
    except Exception as e:
        conn.execute("ROLLBACK")
        raise RuntimeError(f"Historical load failed — DB rolled back. Retryable. Cause: {e}") from e
```

Append-only enforcement: only `INSERT` statements are issued. No `UPDATE` or `DELETE` anywhere in `historical_load.py`.

AC6 compliance: the `except` block's `ROLLBACK` guarantees that a partial load never silently leaves the DB in a mixed state. The raised `RuntimeError` surfaces the error explicitly and confirms retryability.

### Step 7 — Daily Append Operation (AC4, AC3)

```python
def append_day(db_path: str, date: str, spx_daily_high: float,
               spx_12d_ema: float, spx_25d_ema: float, mmth: float) -> None:
    """
    Append a single day's indicator values to the database.
    Uses INSERT OR IGNORE — if the date already exists, the row is silently skipped.
    No existing rows are ever modified or deleted.
    """
    conn = get_connection(db_path)
    conn.execute(
        "INSERT OR IGNORE INTO indicators (date, spx_daily_high, spx_12d_ema, spx_25d_ema, mmth) "
        "VALUES (?, ?, ?, ?, ?)",
        (date, spx_daily_high, spx_12d_ema, spx_25d_ema, mmth)
    )
    conn.commit()
```

`INSERT OR IGNORE` satisfies AC3 (no existing record is modified) and AC4 (new records are appended). Idempotent by design — running the daily append twice for the same date is safe.

## Edge Cases

### AC1 — Initial Historical Load: 1+ Year of Data

- **Happy path:** `yfinance.download('^GSPC', period='2y')` returns ~504 rows; after trim, 252+ rows are stored; MMTH data aligns on the same dates.
- **Boundary:** Market holidays — trading days only; no null rows for non-trading days (yfinance returns only trading days by default).
- **Error path:** yfinance network failure → `RuntimeError` raised, DB rolled back, clear error message with retry instruction.
- **Error path:** MMTH fetch returns fewer than 252 trading days → raise `ValueError` with explicit message before attempting DB write.

### AC2 — Schema Compliance

- **Happy path:** Table created with correct column names and types; `spx_daily_high` column present.
- **Boundary:** Schema creation called twice (idempotent) — `CREATE TABLE IF NOT EXISTS` prevents error.
- **Error path:** DB file path directory does not exist → `sqlite3.OperationalError` raised with clear path reference.

### AC3 — Append-Only Constraint

- **Happy path:** No `UPDATE` or `DELETE` SQL is present anywhere in the codebase (static code review gate).
- **Boundary:** `INSERT OR IGNORE` on a duplicate date — row count unchanged, no exception.
- **Error path:** No error path exists for this AC by design — the constraint is structural (no mutation methods exist to call).

### AC4 — Daily Append

- **Happy path:** New date's values are inserted; `SELECT COUNT(*)` before and after shows +1 row.
- **Boundary:** Append called with today's date when today's data is already present (duplicate) — `INSERT OR IGNORE` skips silently.
- **Error path:** DB file is read-only or locked → `sqlite3.OperationalError` surfaced to caller.

### AC5 — Persistence Across Restarts

- **Happy path:** Row written, connection closed and garbage-collected, new connection opened, row is retrievable.
- **Boundary:** DB file moved to a different path between runs — connection factory receives the new path; rows still present.
- **Error path:** DB file deleted between runs → `sqlite3.OperationalError` on re-open; user must re-run historical load.

### AC6 — Error Handling on Partial Historical Load Failure

- **Happy path:** Full load completes; `COMMIT` succeeds; 252+ rows present.
- **Boundary:** Failure at row 1 of 260 → ROLLBACK; DB has 0 rows; next run starts fresh.
- **Boundary:** Failure at row 259 of 260 → ROLLBACK; DB has 0 rows (not 259); no partial state.
- **Error path:** Network disconnects mid-fetch (before any DB writes) → fetch raises exception; transaction never started; DB is clean.
- **Error path:** DB disk full during INSERT → `sqlite3.OperationalError`; ROLLBACK; 0 rows stored; error surfaced.

## Risk Register

| Risk | Severity | Mitigation |
|------|----------|------------|
| `^MMTH` unavailable on Yahoo Finance | High | Step 2 verification is mandatory before `mmth.py` implementation. EODData fallback is pre-planned. If EODData requires login, surface to human before proceeding. |
| yfinance Yahoo Finance API changes breaking fetch | Medium | Pin `yfinance` version in `requirements.txt`. The library maintainers historically patch API changes within days. For the initial load, a one-time fetch is sufficient even if yfinance is temporarily broken. |
| EMA warm-up insufficient (using exactly 1 year of data) | Medium | Mitigated by design: fetch `period='2y'` and compute EMAs on full 2-year set before trimming to 1-year storage window. Verified in T4. |
| SQLite WAL mode file locking on network/cloud drives | Low | DB file should be stored on a local drive, not a network share. Document this in code comments. WAL creates a `-wal` and `-shm` sidecar file — these must not be deleted while the DB is open. |
| AC6 ROLLBACK silent failure (transaction not started) | Low | Explicitly call `conn.execute("BEGIN")` before any writes. Python `sqlite3` module uses autocommit by default — explicit `BEGIN` ensures the transaction boundary is set before any INSERT. |
| Date alignment gaps (SPX has data on a date where MMTH does not) | Low | Inner-join on dates from both sources before INSERT. Any date with missing data from either source is excluded from storage. Gaps are logged as warnings, not errors. |
| Turn budget overrun on CI remediation | Medium | Implementation is sequential and straightforward. Estimated: ~40 turns for implementation, ~30 turns for test writing, ~30 turns for CI remediation budget. Total ~100 turns, within the 150-turn target. |
