## Approach Evaluation Ruling

Two possible "live fetch" strategies were evaluated:

| Option | Description | Assessment |
|--------|-------------|------------|
| A (selected) | Call existing `fetch_spx(period='2y')` and `fetch_mmth()`; extract `.iloc[-1]` row/last value | Correct. Reuses implemented 2y warm-up. EMA values are accurate. |
| B (rejected) | Call yfinance with `period='1mo'` and compute EMAs independently | Violates DEC-2026-04-22-01. Short window produces incorrect EMA values. No viable alternative. |

Option A is the only correct approach. `approach-evaluation` skill was NOT invoked — this is a mechanical implementation choice, not an architectural decision between viable alternatives.

## Detailed Step Notes

### Step 1 — Module Scaffold and FetchError

`src/fetch/live.py` structure:

```
FetchError(RuntimeError)
  — message must name WHICH indicator(s) failed (AC5)

fetch_live(db_path: str, _spx_df=None, _mmth_series=None) -> dict
  — _spx_df / _mmth_series kwargs allow test injection (follows conftest.py convention)
  — returns dict: {date, spx_daily_high, spx_12d_ema, spx_25d_ema, mmth}
  — raises FetchError on any fetch failure or null value
```

The injectable kwargs `_spx_df` and `_mmth_series` mirror the pattern used in E01S06 tests — they allow unit tests to bypass real network calls without needing `unittest.mock.patch`.

### Step 2 — Fetch Phase

Extraction logic:
- Call `fetch_spx(period="2y")` → `spx_df` (or use `_spx_df` if injected)
- Call `fetch_mmth(period="2y")` → `mmth_series` (or use `_mmth_series` if injected)
- Extract: `row = spx_df.iloc[-1]` → `date`, `spx_daily_high`, `spx_12d_ema`, `spx_25d_ema`
- Extract: `mmth_value = mmth_series.iloc[-1]`
- MMTH date alignment: the last date in `mmth_series` must be within 5 calendar days of the last date in `spx_df`. If the gap exceeds 5 days, treat MMTH as stale and raise `FetchError("MMTH data is stale — last available date is {date}")`.

Each fetch call must be wrapped in its own try/except:
- SPX fetch failure → `FetchError("SPX fetch failed: {cause}")`
- MMTH fetch failure → `FetchError("MMTH fetch failed: {cause}")`
- Both fail → `FetchError("SPX fetch failed: {cause1}; MMTH fetch failed: {cause2}")`

This satisfies AC5: error message names which indicator(s) failed.

### Step 3 — Validation Phase (AC6)

After extraction, check each of the four values:

```
failed = []
if spx_daily_high is None or pd.isna(spx_daily_high): failed.append("spx_daily_high")
if spx_12d_ema is None or pd.isna(spx_12d_ema):       failed.append("spx_12d_ema")
if spx_25d_ema is None or pd.isna(spx_25d_ema):       failed.append("spx_25d_ema")
if mmth_value is None or pd.isna(mmth_value):          failed.append("mmth")

if failed:
    raise FetchError(f"Fetch succeeded but values are null/missing: {failed}. No data persisted.")
```

If `failed` is non-empty, raise `FetchError` and do NOT call `append_day()`. This is the AC6 guarantee: an incomplete indicator set never proceeds to persistence or downstream classification.

### Step 4 — Persistence Phase (AC7)

Only reached if validation passes (all four values are non-null floats):

```python
append_day(
    db_path=db_path,
    date=str(date_value),
    spx_daily_high=float(spx_daily_high),
    spx_12d_ema=float(spx_12d_ema),
    spx_25d_ema=float(spx_25d_ema),
    mmth=float(mmth_value),
)
```

`append_day()` is already idempotent (`INSERT OR IGNORE`). Calling `fetch_live()` twice on the same day is safe — the second call's persist is a no-op on the DB.

`append_day()` raises `RuntimeError` on DB failure. This must propagate unchanged — do not swallow it.

### Step 5 — Test Design

Test file: `tests/test_live_fetch.py`

Pattern: class `TestFetchLive` with `test_T{N}_description` naming. Far-future dates (2099-xx-xx) for DB rows. Inject `_spx_df=sample_spx_df` and `_mmth_series=sample_mmth_series` to bypass network.


## Edge Cases

### AC1 / AC2 / AC3 (SPX fetch — spx_daily_high, spx_12d_ema, spx_25d_ema)

- **Happy path:** `fetch_spx()` returns DataFrame with 252 rows; `.iloc[-1]` yields today's (or most recent trading day's) values; all three floats are non-null and positive.
- **Boundary — weekend/holiday:** `fetch_spx()` returns the most recent trading day; `date` in the returned row may be Friday on a Monday call. This is expected and correct — `append_day()` uses `INSERT OR IGNORE`, so re-running on the same trading day is safe.
- **Boundary — market not yet open (pre-market call):** yfinance may return yesterday's data as the "latest". This is acceptable — the value reflects the most recent complete trading day.
- **Error path — yfinance network failure:** `fetch_spx()` raises `RuntimeError`; caught and re-raised as `FetchError("SPX fetch failed: ...")`. `append_day()` is NOT called.
- **Error path — empty DataFrame:** `fetch_spx()` raises `RuntimeError` internally (existing guard). Propagated as above.

### AC4 (MMTH fetch)

- **Happy path:** `fetch_mmth()` → EODData path (DEC-2026-04-22-02) → Series with 400+ values; `.iloc[-1]` is a float in [0, 100].
- **Boundary — EODDATA_API_KEY not set:** `fetch_mmth()` raises `RuntimeError` with clear instructions; caught and re-raised as `FetchError("MMTH fetch failed: ...")`.
- **Boundary — EODData returns stale data (last date > 5 days ago):** detected in Step 2 staleness check; raises `FetchError("MMTH data is stale — last available date is {date}")`.
- **Error path — EODData HTTP error:** `requests.HTTPError` raised by `fetch_mmth()`; caught and re-raised as `FetchError`.
- **Error path — both yfinance and EODData fail:** `fetch_mmth()` raises combined `RuntimeError`; wrapped in `FetchError` naming MMTH.

### AC5 (fetch failure error handling)

- **Happy path:** Not applicable (AC5 is an error path AC).
- **Error path — one indicator fails:** `FetchError` message explicitly names the failed indicator (e.g., "MMTH fetch failed: ..."). No other indicator's value is displayed as if the full set were available.
- **Error path — multiple indicators fail:** Single `FetchError` names all failed indicators in one message.
- **Constraint:** The error must be visible. `fetch_live()` raises `FetchError` — the caller (dashboard entrypoint or test) is responsible for displaying it. The plan does not introduce a display layer (out of scope for E01S01).

### AC6 (null value error handling)

- **Error path — fetch succeeds but value is NaN:** `pd.isna()` check catches numpy NaN, Python `None`, and pandas `pd.NA`. `FetchError` names the null field. `append_day()` is not called.
- **Error path — fetch succeeds but value is string "null" or similar:** `float()` conversion is applied in Step 4 before `append_day()` — this will raise `ValueError` if the value is non-numeric. The validation in Step 3 uses `pd.isna()` which handles `None` and `NaN`; a non-numeric string would not be caught by `pd.isna()`. Mitigation: after `float()` conversion in Step 3, also check `math.isfinite()` to catch `inf` and `-inf` values.
- **Invariant:** No regime classification proceeds from an incomplete set — this is enforced structurally because `fetch_live()` raises before returning, and callers receive either a complete dict or a `FetchError`.

### AC7 (DB persistence)

- **Happy path:** `append_day()` succeeds; row is retrievable from DB with matching values.
- **Boundary — same date called twice:** `INSERT OR IGNORE` — second call is a no-op. DB row count unchanged. No error raised.
- **Boundary — DB does not exist yet:** `append_day()` calls `create_schema()` internally (idempotent) — schema is created if not present. DB file is created by SQLite on first open.
- **Error path — DB file is read-only or locked:** `append_day()` raises `RuntimeError`; propagated from `fetch_live()` unchanged (not wrapped in `FetchError` — this is a DB error, not a fetch error).
- **Append-only guarantee:** `append_day()` uses only `INSERT OR IGNORE`. No `UPDATE` or `DELETE`. Historical data is never modified.

## Risk Register

| Risk | Severity | Mitigation |
|------|----------|------------|
| `EODDATA_API_KEY` not set in CI environment | High | Test suite must mock `fetch_mmth()` entirely — no real EODData call in tests. CI environment does not need the key. Add to CI setup docs if a live integration test is ever added. |
| `fetch_spx()` returns a MultiIndex DataFrame (yfinance version variance) | Medium | Already handled inside `fetch_spx()` — the existing flatten guard covers this. No action needed in `live.py`. |
| MMTH date lag behind SPX date (EODData updates later in the day) | Medium | 5-calendar-day staleness threshold (Step 2). If MMTH lags by 1 trading day, it is still within threshold and accepted. Document this tolerance in code comments. |
| `float()` conversion of NaN produces `nan` (not caught by `pd.isna()` pre-conversion) | Low | Apply `pd.isna()` BEFORE `float()` conversion. Also add `math.isfinite()` post-conversion to catch `inf`/`-inf`. |
| Test injection pattern (`_spx_df` kwarg) conflicts with future refactor | Low | The `_spx_df` / `_mmth_series` injection pattern is already established in E01S06 conftest. No conflict. |
| Turn budget: implementation is simple but AC5/AC6 error paths require thorough testing | Low | Estimated ~25 turns for implementation, ~20 turns for test writing, ~30 turns CI remediation budget. Total ~75 turns — well within 150-turn target. |
| `append_day()` raises `RuntimeError` on DB failure — must not be silently swallowed | Medium | Explicitly documented in Step 4: `RuntimeError` from `append_day()` propagates unchanged. Test T3 verifies persistence succeeds under normal conditions. |
