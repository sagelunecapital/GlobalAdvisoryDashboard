## Approach Evaluation (approach-evaluation)

### Problem Statement

E01S02 requires detecting a "prior swing high" and "prior swing low" from the `spx_daily_high` time series stored in SQLite. The story explicitly delegates algorithm definition to Delivery. Three viable approaches exist with meaningfully different complexity, dependency, and determinism profiles. The choice of algorithm affects correctness of the anchor, accuracy of divergence classification, and testability.

### Evaluation Criteria

| # | Criterion | Weight | Source |
|---|-----------|--------|--------|
| C1 | Stack compatibility — works with Python + SQLite + pandas (no new runtime deps) | must-have | DEC-2026-04-22-01 |
| C2 | Determinism — same inputs always produce same anchor date | must-have | DEC-2026-04-18-02 (anchor must be stable across the same run) |
| C3 | No tuning parameters that alter QA behavior unexpectedly | important | Story notes: "QA verifies against Delivery's documented algorithm" |
| C4 | Testability with synthetic DB data (injectable test data, tmp_db fixture) | must-have | Story + conftest.py pattern |
| C5 | Operational simplicity — maintainable by a single developer | important | Project context: solo fund manager |
| C6 | Correctness for the specific domain (divergence detection over weeks/months, not milliseconds) | must-have | AC1, AC2 — story example spans Jan–Apr 2026 (87 days) |

### Approaches Identified

**Approach A — Fixed Lookback Window**

Mechanism: Query all `spx_daily_high` rows in the N calendar days immediately before the current date. Prior swing high = `MAX(spx_daily_high)` in that window. Prior swing low = `MIN(spx_daily_high)` in that window. N is a module-level constant documented in code (`SWING_LOOKBACK_DAYS = 90`). The anchor date is the date on which that max/min occurred.

Evidence: Widely used in quantitative finance for simple regime detection when the exact peak morphology is less important than identifying a representative prior level. Pandas `rolling().max()` and SQL `MAX()` are standard, well-documented operations. No exotic dependencies.

Limitations: A 90-day window may miss a swing high/low that occurred 91 days ago if that level is more "meaningful" by a human assessment. The window is arbitrary but documented and fixed — QA can verify deterministically. On very flat markets the max and the current value may be identical (handled: same-day anchor means the run never started, so NO_DIVERGENCE is returned).

**Approach B — Local Peak Detection via scipy argrelextrema**

Mechanism: Load full SPX series; apply `scipy.signal.argrelextrema(data, np.greater, order=N)` to identify local maxima (N = minimum gap between peaks in bars). Prior swing high = the most recent local maximum before the current date.

Evidence: `argrelextrema` is the canonical scipy tool for local extrema detection. Used in academic backtesting literature. Identifies true morphological peaks rather than absolute max in a window.

Limitations: Requires `scipy` as a new runtime dependency — violates C1 spirit (DEC-2026-04-22-01 enumerates the stack; scipy is not listed). The `order` parameter is a tuning parameter that controls peak sensitivity — violates C3 (QA can only verify against the documented parameter value, and changing it changes all anchor dates). Requires loading the full historical series for every call (performance concern at scale, though minor for this use case).

**Approach C — Percentage-Based Threshold**

Mechanism: Walk backward from the current date until `spx_daily_high` has dropped ≥ X% from a running maximum. The date of that running maximum is the prior swing high. X is a tuning parameter (e.g., 5%).

Evidence: Used in trend-following systems as a simplified swing definition ("significant retracement"). Adaptive to market volatility in a loose sense.

Limitations: Introduces a percentage threshold tuning parameter (C3 violation — even more sensitive than scipy's `order`). Directionally biased: the walk-backward is well-defined for highs but becomes ambiguous for swing lows (walk-backward from a low looking for a ≥X% recovery is not symmetric). Non-deterministic at boundary values. Hardest to explain in QA.

### Comparison Matrix

| Criterion | A: Fixed Window | B: scipy argrelextrema | C: Pct Threshold |
|-----------|----------------|----------------------|-----------------|
| C1: Stack compatibility | PASS — pure pandas + SQL | FAIL — adds scipy dep | PASS — pure pandas |
| C2: Determinism | PASS — MAX() is deterministic | PASS (given fixed order param) | PARTIAL — float precision edge cases |
| C3: No surprise tuning params | PASS — one documented int constant | FAIL — order param changes all anchor dates | FAIL — pct param changes all anchor dates |
| C4: Testability | PASS — inject rows, assert MAX date | PASS — but harder to construct minimal test cases | PARTIAL — threshold boundary cases are fiddly |
| C5: Operational simplicity | PASS — 5 lines of SQL/pandas | PARTIAL — scipy API requires familiarity | FAIL — non-obvious walk-backward logic |
| C6: Domain correctness | PASS — story example spans 87 days; 90d window captures it | PASS — more "correct" peak morphology | PARTIAL — symmetric definition breaks for lows |

### Trade-offs

**Approach A (Fixed Window)**
- Gains: Zero new dependencies; fully deterministic; QA-verifiable; matches the story's example (87-day span fits in 90-day window); simple to test with synthetic data.
- Costs: May miss a more "meaningful" peak outside the 90-day window. The 90-day constant is a definition, not a discovery — it must be documented clearly.
- Lock-in: The window constant is documented in the module and in this plan. Changing it is a one-line code change with full test re-verification. Low lock-in.

**Approach B (scipy)**
- Gains: Identifies true morphological peaks; closer to how a human analyst would draw swing levels.
- Costs: New runtime dependency; `order` parameter instability; harder to construct minimal synthetic tests; violates the stack constraint.
- Lock-in: Once scipy is in the dependency tree, it affects packaging, CI, and future contributors.

**Approach C (Pct Threshold)**
- Gains: Adaptive without a hard calendar window.
- Costs: Non-symmetric for lows; two tuning parameters in practice; hardest to reason about for QA.
- Lock-in: The threshold becomes an implicit assumption baked into every historical classification.

### Ruling

**Approach A selected.** It is the only approach that satisfies all must-have criteria (C1, C2, C4, C6) without introducing a new dependency or an unstable tuning parameter. The 90-day window constant is set to 90 because the story's worked example spans 87 calendar days (Jan 28 to Apr 15, 2026) — the window is wide enough to cover the example definitively. The constant is named `SWING_LOOKBACK_DAYS` and documented in the module docstring. QA verifies against this documented value.

## Detailed Step Notes

### Step 1 — DivergenceResult Enum

Define in `src/analysis/divergence.py`:

```
class DivergenceResult(Enum):
    BEARISH = "BEARISH"
    BULLISH = "BULLISH"
    NO_DIVERGENCE = "NO_DIVERGENCE"
    DATA_GAP = "DATA_GAP"
```

`DATA_GAP` is a first-class enum member. It is structurally impossible for a downstream consumer to accidentally treat `DATA_GAP` as `NO_DIVERGENCE` — they are different enum values that fail equality checks against each other. This satisfies AC5's requirement that DATA_GAP be "distinct from NO_DIVERGENCE."

### Step 2 — Function Signature and Module Contract

Full signature:

```
SWING_LOOKBACK_DAYS: int = 90   # module-level constant

detect_divergence(
    db_path: str,
    as_of_date: str,       # ISO-8601 YYYY-MM-DD; the date being evaluated
    _db_rows=None          # injectable for tests: list of (date, spx_daily_high, mmth) tuples
) -> tuple[DivergenceResult, str]
```

The injectable `_db_rows` kwarg follows the pattern established in E01S01's `_spx_df` / `_mmth_series`. When `_db_rows` is not None, the function uses that list directly and skips the DB read. This allows tests to pre-populate known data without using tmp_db for every case — though tests that verify DB read behavior (AC4) must still use tmp_db.

Return contract (documented in docstring):
- `(DivergenceResult.BEARISH, str)` — bearish divergence detected; str is human-readable description
- `(DivergenceResult.BULLISH, str)` — bullish divergence detected
- `(DivergenceResult.NO_DIVERGENCE, str)` — no divergence condition met
- `(DivergenceResult.DATA_GAP, str)` — insufficient history; str contains user-visible explanation; downstream consumers MUST treat this as unavailable input, not clean no-divergence

The function does NOT raise on DATA_GAP — it returns it. This is intentional: DATA_GAP is an expected operational state when the DB is newly populated, not an error condition.

### Step 3 — DB Read and DATA_GAP Detection

DB read logic:

```
SELECT date, spx_daily_high, mmth
FROM indicators
WHERE date < :as_of_date
ORDER BY date ASC
```

After loading rows into a DataFrame:

1. Check if the most recent row date (last row's date) is at least `SWING_LOOKBACK_DAYS` calendar days before `as_of_date`. If NOT, return DATA_GAP.
   - Specifically: if `(pd.Timestamp(as_of_date) - pd.Timestamp(df.iloc[-1]["date"])).days < SWING_LOOKBACK_DAYS`, return DATA_GAP.
   - Also: if the DataFrame has zero rows, return DATA_GAP immediately.

2. Slice the lookback window: `window_df = df[df["date"] >= window_start_date]` where `window_start_date = (pd.Timestamp(as_of_date) - pd.Timedelta(days=SWING_LOOKBACK_DAYS)).strftime("%Y-%m-%d")`.

3. Also retrieve the current date's row for `as_of_date` itself (the row being evaluated):
   ```
   SELECT spx_daily_high, mmth FROM indicators WHERE date = :as_of_date
   ```
   If this row does not exist, return DATA_GAP ("No data for as_of_date {as_of_date}").

Note on the current date's MMTH: per DEC-2026-04-18-02, the divergence check compares the CURRENT date's MMTH against the MMTH recorded ON the prior swing anchor date. Both values come from the `indicators` table.

### Step 4 — Swing Anchor Detection

From `window_df` (rows strictly before `as_of_date` within the 90-day window):

```
prior_high_idx = window_df["spx_daily_high"].idxmax()
prior_high_date = window_df.loc[prior_high_idx, "date"]
prior_high_spx  = window_df.loc[prior_high_idx, "spx_daily_high"]
prior_high_mmth = window_df.loc[prior_high_idx, "mmth"]

prior_low_idx  = window_df["spx_daily_high"].idxmin()
prior_low_date = window_df.loc[prior_low_idx, "date"]
prior_low_spx  = window_df.loc[prior_low_idx, "spx_daily_high"]
prior_low_mmth = window_df.loc[prior_low_idx, "mmth"]
```

These four anchor points are stable for all days in the same run (DEC-2026-04-18-02). The function does NOT attempt to track "which run" across calls — each call computes the anchor independently from the 90-day window. This is correct: if today is within a bull run, the 90-day window will consistently find the same prior swing low as the anchor until a new, lower low displaces it.

Edge case — ties in `idxmax()` or `idxmin()`: pandas `idxmax()` returns the first occurrence of the maximum by default. This is acceptable and documented — if two dates share the same maximum `spx_daily_high`, the earlier date is the anchor. This is deterministic and QA-verifiable.

### Step 5 — Bearish Divergence Check (AC1)

Conditions (both must be true):
1. `current_spx_high > prior_high_spx` — SPX has exceeded the prior swing high
2. `current_mmth < prior_high_mmth` — MMTH at current date is lower than MMTH at the prior swing high date

If both true: return `(DivergenceResult.BEARISH, f"Bearish divergence: SPX {current_spx_high:.2f} > prior high {prior_high_spx:.2f} on {prior_high_date}; MMTH {current_mmth:.2f} < {prior_high_mmth:.2f} at anchor")`

Anchor persistence (DEC-2026-04-18-02): Because the function computes from the 90-day window on each call, if today's SPX is a new high within the same run, the anchor remains the same prior swing high (since yesterday's high within the same run is NOT in the lookback window as the maximum — the true prior high is). This is correct by the window design: the anchor is always the max of the prior 90 days, not the max of the current run.

### Step 6 — Bullish Divergence Check (AC2)

Conditions (both must be true):
1. `current_spx_high < prior_low_spx` — SPX has fallen below the prior swing low
2. `current_mmth > prior_low_mmth` — MMTH at current date is higher than MMTH at the prior swing low date

If both true: return `(DivergenceResult.BULLISH, f"Bullish divergence: SPX {current_spx_high:.2f} < prior low {prior_low_spx:.2f} on {prior_low_date}; MMTH {current_mmth:.2f} > {prior_low_mmth:.2f} at anchor")`

Note on SPX direction for bullish: per DEC-2026-04-18-01, `spx_daily_high` is always the daily HIGH — even when detecting a "new low." The prior swing low is still the MINIMUM of `spx_daily_high` values in the window (the least-bad daily high across the drawdown period). This is consistent with the story definition: "SPX makes a new low" means the current daily high is below the prior period's minimum daily high.

### Step 7 — Fallthrough (AC3)

If neither bearish nor bullish condition is met (including the case where SPX is at a new high but MMTH is also higher, or SPX is at a new low but MMTH is also lower), return `(DivergenceResult.NO_DIVERGENCE, "No divergence: SPX and MMTH are not in divergence as of {as_of_date}")`.

## Edge Cases

### AC1 — Bearish Divergence

- **Happy path:** Current `spx_daily_high` > prior window max AND current `mmth` < `mmth` at that max date → BEARISH returned with correct anchor dates in description.
- **Boundary — SPX exactly equals prior high (not strictly greater):** `current_spx_high == prior_high_spx` → condition 1 is False → NO_DIVERGENCE. The story says "makes a new high" meaning strictly greater. Use `>` not `>=`.
- **Boundary — MMTH exactly equals prior anchor MMTH:** `current_mmth == prior_high_mmth` → condition 2 is False (requires strictly lower) → NO_DIVERGENCE. Use `<` not `<=`.
- **Boundary — SPX new high but MMTH also higher:** Condition 2 fails → NO_DIVERGENCE (no divergence exists; price and breadth confirm each other).
- **Error path — current date row missing from DB:** DATA_GAP returned (Step 3 guard).
- **Error path — all MMTH values in window are NaN/None:** Covered by DATA_GAP logic if window is insufficient; if window exists but MMTH is null, this is a DB corruption — the function will raise a pandas error. Not silently returning NO_DIVERGENCE is the correct behavior; the caller sees the exception and can escalate.

### AC2 — Bullish Divergence

- **Happy path:** Current `spx_daily_high` < prior window min AND current `mmth` > `mmth` at that min date → BULLISH.
- **Boundary — SPX exactly equals prior low:** `current_spx_high == prior_low_spx` → condition 1 is False → NO_DIVERGENCE. Use `<` strictly.
- **Boundary — SPX new low but MMTH also lower:** Condition 2 fails → NO_DIVERGENCE (no divergence; breadth confirms the price decline).
- **Boundary — prior high and prior low are the same row (flat market or single-row window):** Both anchor dates are the same date. Bearish and bullish checks operate on different anchors but from the same row. If current SPX is both above the max and below the min simultaneously — mathematically impossible. The two checks are evaluated in order (bearish first); if somehow both could trigger, bearish takes precedence (defined by check order, documented).

### AC3 — NO_DIVERGENCE

- **Happy path:** SPX is within the prior high/low range → NO_DIVERGENCE.
- **Boundary — SPX is at a new high AND MMTH is also at a new high relative to the anchor:** Both conditions for bearish fail on condition 2 → NO_DIVERGENCE. This is "confirmed bullish momentum," not divergence.
- **Boundary — empty window after slicing (window_df is empty but full df is not):** Cannot happen if DATA_GAP check passes (at least 90 days of data exist). If window_df is somehow empty after the check, raise `ValueError` — this is a programming error, not a DATA_GAP state.

### AC4 — Reads from Historical DB

- **Happy path:** DB contains 400+ rows (E01S06 guarantee of ≥1 year); query returns correctly ordered DataFrame; anchor computed correctly.
- **Boundary — DB has exactly 90 days of rows:** Just passes the DATA_GAP threshold; function proceeds. Window includes all 90 rows.
- **Boundary — DB has 89 days of rows:** Returns DATA_GAP.
- **Error path — DB file does not exist:** `sqlite3.OperationalError` raised by `get_connection()`; propagates from `detect_divergence()` unchanged (not swallowed).
- **Error path — DB file exists but `indicators` table missing:** `sqlite3.OperationalError` on the SELECT; propagates unchanged.

### AC5 — DATA_GAP

- **Happy path:** DATA_GAP returned with human-readable explanation string.
- **Boundary — DATA_GAP vs NO_DIVERGENCE identity check:** `DivergenceResult.DATA_GAP != DivergenceResult.NO_DIVERGENCE` is always True (enum members are distinct). Downstream consumers using `if result == DivergenceResult.NO_DIVERGENCE` will NOT accidentally match DATA_GAP.
- **Boundary — DATA_GAP explanation string is non-empty:** Assert `len(explanation) > 0` in test_T6, test_T7, test_T8.
- **Error path — as_of_date row exists in DB but zero prior rows exist:** Returns DATA_GAP immediately (zero rows before as_of_date).
- **Constraint:** DATA_GAP is returned, not raised. Callers must handle it explicitly in their control flow.

## Risk Register

| Risk | Severity | Mitigation |
|------|----------|------------|
| `idxmax()` ties (two dates with identical `spx_daily_high`) — anchor is non-unique | Low | Documented: first occurrence (earlier date) is selected. Test_T1 uses rows with a unique maximum. No special handling needed — determinism is preserved. |
| `as_of_date` falls on a weekend/holiday with no DB row (common for daily data) | Medium | Step 3: if no row exists for `as_of_date`, return DATA_GAP. Callers must query using the most recent trading date, not a calendar date. Document in function docstring. |
| 90-day window excludes a meaningful swing high/low from 91+ days ago | Low | The 90-day constant is a definition, not a claim of completeness. Divergence detection operates on this defined window. QA verifies against the documented algorithm, not against human-assessed "meaningful" peaks. |
| Both bearish AND bullish conditions cannot simultaneously be true (SPX cannot be both above prior high and below prior low) | Info | Mathematically impossible unless the window has only one row where max == min. If prior_high == prior_low (flat market), SPX cannot strictly exceed AND fall below the same value. Check order (bearish first) documents priority; the scenario is unreachable in practice. |
| MMTH is NULL for the prior anchor date in the DB (data quality issue) | Medium | If `prior_high_mmth` is NaN, the comparison `current_mmth < NaN` is False in pandas — would incorrectly return NO_DIVERGENCE. Mitigation: after computing anchors, assert `pd.notna(prior_high_mmth)` and `pd.notna(prior_low_mmth)`; if either is NaN, return DATA_GAP with explanation "MMTH missing for anchor date {date}". |
| Test injection (`_db_rows`) bypasses `get_connection()` entirely — DB path not validated in injection mode | Low | Acceptable — injection is a testing affordance, not a production path. Document: when `_db_rows` is provided, `db_path` is ignored. |
| Turn budget: swing detection is deterministic but test matrix is broad (12 tests) | Low | Estimated ~20 turns implementation, ~25 turns test writing, ~30 turns CI remediation. Total ~75 turns — well within 150-turn target. |
