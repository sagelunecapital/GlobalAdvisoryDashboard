# Epic: GAAI Daemon Windows Robustness

## Purpose

The GAAI delivery daemon is the engine that autonomously delivers backlog stories. On Windows — the primary development environment — both daemon launcher scripts (`daemon-start.sh:35-40`, `delivery-daemon.sh:126-135`) unconditionally exit before any delivery can begin. Usage-limit hits orphan `in_progress` stories with no resume path. As a result, delivery sessions require full human orchestration on Windows, defeating the purpose of the daemon.

This Epic delivers a Python-based delivery entrypoint (`gaai_deliver.py`) that runs natively on Windows, a resumable SQLite job queue, a pre-flight doctor command, and a Windows Task Scheduler–compatible `--unattended` mode — so stories can be delivered overnight without supervision.

---

## Scope

- `gaai_deliver.py` — Python entrypoint that replaces the bash launcher on Windows; delegates to existing bash daemon on Linux/macOS
- `data/gaai-jobs.db` — SQLite job queue for resumable delivery state across usage-limit hits and process kills
- `gaai doctor` (`gaai_deliver.py --doctor`) — pre-flight command that auto-fixes known Windows environment issues
- `--unattended` flag and `gaai-deliver.bat` wrapper for Windows Task Scheduler cron-compatible scheduling
- Integration tests simulating each documented failure mode

---

## Out of Scope

- Modifying `delivery-daemon.sh` or `daemon-start.sh` (Linux/macOS daemon is unchanged)
- Cloud daemon, distributed execution, or multi-project support
- Changes to Story generation logic or backlog lifecycle state definitions (adding/removing states from the state machine) — the `in_progress → refined` orphan recovery authorized by DEC-2026-05-12-02 is a governed exception to this, not a contradiction
- WSL as a supported primary path (WSL remains a detected fallback only)
- UI changes of any kind
